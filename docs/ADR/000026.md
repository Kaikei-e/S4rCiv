# ADR-000026: 議事録の読書目次を発言順インデックスに限定し発言者ロスター化を禁じる

## Status

Accepted

## Date

2026-06-10

## Affected Services

- web (SvelteKit) — `DocOutline.svelte`（読書目次コンポーネント）は議事録に対して**発言順インデックス**だけを描画する。`meetings/[issueId]/+page.svelte` は各発言に `sp-{発言順}` アンカーを振り、目次項目は発言と 1:1 で発言順に並ぶ。法令側（`laws/[lawId]`）の条目次は別問題で eId 由来アンカーを使う。

## Context

長い議事録（会議録）に、本文へジャンプするための読書目次（デスクトップ左ガターの ToC、≥1200px で表示）を新設した。法令の条目次は「第N条」を並べるだけで素直だが、議事録の目次は**単位の選び方次第で設計不変条件に抵触する**。

議事録のデータは発言（speech）の列で、各発言は発言者名・会派・発言順を持つ。目次を作る際、発言者を名寄せして「この人の発言へ」と飛べる**ロスター型目次**は一見便利である。しかしこれは特定議員の発言だけを集めて辿れる導線＝**個人別の発言集成軸**を新たに作ることになり、二つの線に同時に触れる。

- **著作権法40条1項但書**（同一の著作者のものを編集して利用する場合）。[[000004]] が確定したとおり、議員発言の全文表示は40条1項本文で正当化されるが、特定議員の発言だけを集成するアンソロジーは但書で権利制限の適用外となる。
- **DISCIPLINE §4**（反 doxxing・横断名寄せ禁止）。提示軸は public actor の事実記録に限り、私人を含めた個人プロファイルを作らない。

[[000004]] は既に「個人別『発言集成』UI を作らない／提示軸は **会議 / 時系列 / 差分のみ**」を設計不変条件として固定している。今回追加した読書目次は新しい UI サーフェスであり、ここでロスター化の誘惑が再び生じる。将来「Xの発言へ」をワンクリックで足したくなるのは自然な UX 改善に見えるが、それが [[000004]] 違反であることは**目次のコードだけを読んでも自明でない**。この非自明な境界を、目次 UI に対して明示的に固定する必要がある。

本決定は passive / read-only、plane 分離（目次は解釈面 read model 上の純ナビゲーションで新しい事実を作らない）、source compliance（出典 permalink ＋取得時刻の常時併記は変えない）と整合する。

## Decision

**議事録の読書目次は発言順インデックスに限定し、発言者ロスター化（per-person 集成軸）を禁じる。[[000004]] の「個人別発言集成 UI を作らない／提示軸は 会議 / 時系列 / 差分のみ」を、読書目次 UI へ拡張適用する。**

1. **目次は発言順（会議軸）を保つ。** 目次項目は本文の発言と **1:1**・発言順に並ぶ。`DocOutline` は渡された順序をそのまま描画し、並べ替え・グルーピングをしない。
2. **同一話者も束ねない。** 同じ発言者名が何度現れてもラベルにすぎず、名寄せ・重複排除をしない。「この発言者の発言だけ集める」導線は作らない。
3. **アンカーは `sp-{発言順}`**（会議内で安定）。可視ラベルは発言者名＋会派（可読）で、アンカーのフラグメントは内部ナビゲーション用ハンドルであって一次ラベルではない（[[000021]] の「内部識別子を一次ラベルに出さない」と整合）。記録 #seq の引用ハンドルとは別系統・非競合。
4. **法令側は対象外。** 法令の条目次は per-person 軸の問題を持たないため eId 由来アンカー（`n-{eId}`）で別に扱う。

検討した代替案と却下理由:

- **発言者ロスター目次（名寄せして「この人の発言へ」）**: 長い議事録のナビとして便利だが、40条1項但書＋ DISCIPLINE §4 に触れる個人別集成軸を新設してしまう。→ 却下（[[000004]] の設計不変条件に正面から反する）。
- **議事録には目次を出さない**: ロスター化リスクは確実に消えるが、長い会議録での発言ジャンプという本来の不便を解消できない。→ 却下（発言順インデックスなら原則を守ったまま不便を解消できる）。
- **目次を出すが本文同様「会議軸」と明記せず実装に委ねる**: 将来の改修でロスター化が忍び込む余地を残す。→ 却下（非自明な境界なので決定として固定する）。

## Consequences

### Pros

- 長い議事録でも発言にジャンプでき、UX を確保しつつ [[000004]] の設計不変条件を保てる。
- 「目次は発言順・1:1・話者で束ねない」という単純な不変条件になり、40条但書（法）と反 doxxing（規律）を同時に満たす単一ルールが目次 UI にも一本化される。
- コードコメント（`DocOutline.svelte` / `meetings/[issueId]/+page.svelte`）とテスト（発言順保持・同一話者非グルーピングをピン留め）で回帰が落ちる。

### Cons / Tradeoffs

- 「発言者ロスターを作らない」を目次 UI でも恒常的に守る規律コストが残る（発言者を属性として表示・リンクするのは可だが、特定個人の発言を集成する目次は出さない、という線引きの徹底が要る）。`speaker` から記名投票記録へのリンク（[[000006]] の唯一許す per-person 軸）は従来どおり可。
- 法解釈は現行法・現行規約（2026-06 時点）に依存する。著作権法・NDL 規約の改定時は [[000004]] とともに再評価が要る。
- 目次は ≥1200px のデスクトップのみ表示で、モバイル・狭幅デスクトップでは発言ジャンプ手段を持たない（戻るボタンのみ）。発言数が多い会議録のモバイル・ナビは将来課題として残す。
