# ADR-000015: projection を poll から分離し catch-up subscription として独立ループで回す

## Status

Accepted

## Date

2026-06-05

## Affected Services

- collector (Go) — `cmd/collector` runDaemon（poll ループと project ループを2 goroutine に分離、in-process wake シグナル）
- collector (Go) — `pipeline` interface（`cycle()` を `poll()` と `project()` に分割）

## Context

read model（`interpretation.meeting` ほか）が、新データ取得後に手動 `reproject` 無しでは更新されないように見える事象を調査した。

原因は projection の**起動タイミング**。旧 `runDaemon` は `cycle() = PollOnce(batchSize=100) → projector.Run()` と poll と project を直列に呼んでいた。`projector.Run` は offset 以降の未投影イベントを全 fold する正しい catch-up エンジンだが、poll の後ろに連結されているため、

- **定常運用**: due watch は毎サイクル数件なので poll は即終了し、project が毎サイクル（~60s）走る → 問題なし。
- **大量 backlog**（catch-up discover 直後・初回デプロイ・長期停止明け）: `PollOnce(100)` がソース別レート制限（30s/件）で約50分回り、その間 project がブロックされる → fetch 済みイベントが `observation.event`（snapshot 付き）に在るのに read model に出ない。

これは欠陥というより設計弱点。Web 調査の結果、CQRS/event-sourcing の定石は **projection を取込側から非同期に分離**し、checkpoint ベースで「catch-up フェーズ（溜まりをバッチ）→ live フェーズ（新着を即時）」を回す **catch-up subscription**（[event-driven.io](https://event-driven.io/en/projections_and_read_models_in_event_driven_architecture/)、[Microsoft Learn: Event Sourcing](https://learn.microsoft.com/en-us/azure/architecture/patterns/event-sourcing)）。本プロジェクトの projector は既に checkpoint（`interpretation.projector_offset`）＋ idempotent upsert（reproject 安全、global `seq` 順）で、正しさ要件は満たしている。足りないのは独立した実行ループだけだった。

PostgreSQL `LISTEN/NOTIFY` をトリガに使う案は単一自己ホストバイナリには不適と判断した: 通知は**非永続**（listener 再起動/切断中の通知を取りこぼし恒久的不整合）、8KB ペイロード上限、~1k/s で頭打ち、tx-pooling PgBouncer で破綻（[recall.ai](https://www.recall.ai/blog/postgres-listen-notify-does-not-scale)、[PostgreSQL NOTIFY docs](https://www.postgresql.org/docs/current/sql-notify.html)）。単一プロセスならプロセス内シグナルで十分かつ堅牢。

## Decision

**poll と project を独立した2ループに分離する**（catch-up subscription の単一バイナリ版）。`pipeline` interface の `cycle()` を `poll()`（観測イベント append、emit 件数を返す）と `project()`（read model fold [+ differ]）に分割し、`runDaemon` を2 goroutine で回す:

- **poll ループ**: 日次 autoDiscover（ADR-000012）＋ due watch の PollOnce。イベントを emit したら project ループを**プロセス内 wake チャネル（バッファ1・非ブロッキング送信）で起こす（live フェーズ）**。
- **project ループ**: 起動時に1回 `project()`、以後 **wake（新着）または ticker（`projectInterval`=10s、catch-up／安全網）**で `project()`。projector は checkpoint＋idempotent なので、長い poll と**並行**に走らせても安全に収束する。進行中の poll が逐次 append するイベントを10s毎に fold するため、backlog 中も read model が止まらない。

`projectInterval` を read-model レイテンシの上限とし、poll batch 長に依存させない。`LISTEN/NOTIFY` は採らない。

### 検討した代替案

- **現状維持（直列）**: 定常時は無害だが backlog 時に最大~50分の projection 停止。却下。
- **LISTEN/NOTIFY トリガ**: 通知非永続による取りこぼし＝不整合、PgBouncer 破綻等。単一プロセスにオーバースペック。却下。
- **poll batch 縮小（100→10）**: cycle を早く返すだけの応急策。coupling は残り、巨大 backlog では依然直列化。却下。

## Consequences

### Pros

- read-model レイテンシが poll batch 長から独立し、`projectInterval`（~10s）で上限化。backfill 中も新データが数〜十数秒で UI に反映され、手動 reproject が不要になる。
- live フェーズ（wake）で定常時は即時投影、catch-up フェーズ（tick）で backlog と取りこぼしを必ず回収。
- projection の正しさ要件（checkpoint＋idempotent）は既存のまま。global `seq` 順で順序保証。
- `LISTEN/NOTIFY` の非永続・スケール・プーリング問題を回避。

## Cons / Tradeoffs

- daemon が2 goroutine になり、並行性（poll と project が同一 pgxpool・同一 gateway を共有）の考慮が要る。安全性は「poll は observation/control 表へ書き込み・project は interpretation 表へ書き込みで競合せず、gateway の Fetch(HTTP) と Parse(純粋) は共有可変状態を持たない」ことに依存（`go test -race` 通過）。
- project ループの常時 tick（10s）で、空でも軽い `EventsSince` クエリが走る（`projected==0` はログ抑制済み）。
- 手動 `reproject`（別プロセス）と daemon の project ループが同時に走ると read model 書き込みが競合し得る（既存の運用上の注意。reproject は単発・低頻度）。
