# ADR-000012: 定期 discover を poll daemon に統合し監視リストを全 source で自動追従させる

## Status

Accepted

## Date

2026-06-05

## Affected Services

- collector (Go) — `cmd/collector` runDaemon、`pipeline` interface（`autoDiscover`）、各 pipeline 実装
- collector (Go) — kokkai / egov-law の日付窓 discover、roster / sangiin の固定 discover

## Context

国会会議録（kokkai）のデータが最新まで追随していなかった。調査の結果:

- 監視リスト（control.watch）は第217回国会 / 2025-06-21 までの807件で止まっていた。一方 kokkai API は第221回 / 2026-05-22 まで提供済みで、2025-06-22〜2026-06-05 に634会議録（会期218〜221）が未取得だった。
- 原因はバグではなく**運用の欠落**: poll daemon（`collector run`）は既存 Watch を Poll するだけで、新しい Resource は手動 `discover` を実行しない限り監視リストに入らない。会期217以降 discover が一度も実行されておらず、スケジュール化もされていなかった。
- 同じ構造は egov-law にもある: daemon は既存法令を再 Poll するだけで、新規制定法令は手動 `discover --updated` なしには追従しない。Discover が手動である限り、全 source が「放置すると最新から取り残される」潜在問題を抱える（用語は CONTEXT.md「Watch / Discover / Poll」）。

手動 discover が別プロセスで走るため、DISCIPLINE §1（ソース別 serial＋interval、同一ソースへ同時多発リクエスト禁止）を守るには「discover 中は daemon を止める」運用が必要だった。これも自動追従の妨げだった。

## Decision

**定期 discover を poll daemon に統合する。** `pipeline` interface に `autoDiscover(ctx)` を追加し、`runDaemon` が poll サイクルとは別の長周期（日次）でこれを呼ぶ。各 source は自分の発見方法で実装する:

- **kokkai / egov-law**: 「直近90日」ローリング窓を日次で discover（kokkai=meeting_list、egov=更新法令一覧 `--updated`）。窓は会議日／更新日ベースで、公開遅延（確定版が会議日から数週間〜最大~2ヶ月遅れて出る）と最大90日の daemon 停止を吸収する。upsert は冪等なので再走査は安全、meeting_list はメタデータのみで安価。
- **roster / sangiin**: 既存の固定／ベスポークな discover を日次で再実行（名簿は選挙後に変わり、記名投票は会期内に蓄積するため）。

daemon と discover が**同一プロセス＝同一の直列レートリミッタを共有**するため、DISCIPLINE §1 は自動的に保たれ、in-daemon discover には daemon 停止が不要。手動 `discover` サブコマンド（別プロセス）は従来どおり daemon 停止を要する。

初回の大きな gap（634件）は steady-state のローリング窓では埋まらないため、手動 catch-up discover（`--from 2025-06-22 --until 2026-06-05`）を別途一度だけ実行した。**steady-state＝ローリング窓の自動 discover、長期 gap＝手動 catch-up** と役割を分離する。

### 検討した代替案

- **外部スケジューラ（cron / docker compose run）**: コード変更不要だが別プロセスのため毎回 daemon 停止・再開が要り、§1 のレースと運用の複雑さを招く。却下。
- **専用 discoverer サービス**: poll daemon と同一ホストを叩くためクロスプロセスでレート調整が必要。オーバーエンジニアリング。却下。
- **kokkai のみ対応**: 最小だが egov ほかに同じ潜在バグが残る。`runDaemon` が pipeline interface で全 source 共通な以上、一括対応の方が筋が良い。却下。

## Consequences

### Pros

- 監視リストが全 source で自動的に最新へ追従し、「放置で取り残される」運用バグの類を一掃する（egov の新規制定法令追従も同時に解決）。
- in-daemon discover は直列レートリミッタを共有するため DISCIPLINE §1 を守りつつ daemon 停止が不要。
- 90日窓 ＋ 冪等 upsert により、公開遅延の確定版と中規模の daemon 停止の両方を吸収する。
- steady-state（自動）と長期 catch-up（手動）の責務分離が明確。

### Cons / Tradeoffs

- daemon の責務が「Poll のみ」から「Poll＋定期 Discover」に増える（runDaemon の状態：最終 discover 時刻）。
- 90日は magic number。公開遅延が90日を超える稀な確定版や、90日超の daemon 停止は steady-state では取りこぼし、手動 catch-up が必要（log で可視化する）。
- 日次 discover の meeting_list 走査（~90秒）がそのサイクルの Poll をわずかに後ろ倒しする（日次なので影響軽微）。
- 手動 discover（別プロセス）は引き続き daemon 停止が必要で、運用に2系統が併存する。
