# ADR-000006: M3 の提示軸を横断タイムライン＋議員記名投票に拡張し per-person 軸の線引きを確定する

## Status

Accepted

## Date

2026-06-03

## Affected Services

- api (Go) — `ListTimeline` / `ListLegislatorVotes` RPC を追加。timeline は読取時合成（新 projector/migration なし）、vote は `person_id` 軸の問い合わせ
- db (Postgres) — `interpretation.vote(person_id)` にインデックスを追加。`interpretation.speech(person_id)` は **ADR-000004 の通り意図的に未索引のまま**
- web (SvelteKit) — 全ソース横断タイムライン、議員の記名投票記録ビュー。**発言の個人別「集成」UI は引き続き作らない**

## Context

[[000004]] は「個人別『発言集成』UI を作らない」を**設計不変条件**として固定し、提示軸を**会議 / 時系列 / 差分のみ**に限った。根拠は著作権法40条1項但書（同一著作者の発言を編集して利用するのは権利制限の外）と DISCIPLINE §4（反 doxxing・横断名寄せ禁止）を同時に満たす単一ルールである。

M3（ダッシュボード v0、概念 §11「タイムライン＋議員/キーワードのウォッチ＆アラート」）は、このプロジェクトで**初めて複数ソースを横断する動線**と「議員のウォッチ」を作るマイルストーンである。ここで二つの緊張が顕在化する。

1. **横断タイムラインの person 軸**: タイムラインは観測ログ全体を時系列・横断で辿る主動線になる。ここに person 軸（話者でフィルタ）を許すと、ある人物の活動を時系列に名寄せでき、[[000004]] / §4 の反 doxxing を実質的に破る経路になる。
2. **「議員ウォッチ」(§11) と発言集成禁止の衝突**: 文字通りの「議員ウォッチ」を発言まで含めて実装すると [[000004]] と正面衝突する。

一方、**記名投票（`VoteEvent` / `Vote`、Popolo voting）は性質が異なる**。①著作物でない事実記録であり40条1項但書の射程外、②国会の記名投票の投票者は全員 Diet 議員＝DISCIPLINE §4 が「監視の正当な矛先」と明示する**説明責任を負う公職者**、③私人 doxxing リスクがない。データモデルにも `interpretation.vote.person_id`（Popolo link）が既に存在する。[[000004]] の「提示軸は会議/時系列/差分のみ」は **speech の文脈で書かれた線引き**であり、vote 軸は当時想定されていなかった。

したがって M3 で「§11 の議員ウォッチをどの形で成立させるか」「タイムラインに person 軸を持たせるか」を、[[000004]] を拡張する形で確定する必要が生じた。

## Context（探索で確認した事実）

- `proto/s4rciv/query/v1/query.proto` の既存6 RPC は単体取得／ソース別のみで、複数ソースを横断する口が無い。
- 各リードモデル行は「自分を生んだ event の `observation_seq`」を持つ（join キーになる）。kokkai マイグレーションは「Read models reference observation only, never each other, so each can be truncated independently」を不変条件として明記している。
- `interpretation.vote` は `vote_event_id` のみ索引（`vote_event_fk_idx`）で `person_id` は未索引。`interpretation.speech(person_id)` も [[000004]] の通り意図的に未索引。

## Decision

**M3 の提示軸に「全ソース横断タイムライン」と「議員別の記名投票記録」を加える。タイムラインには person 軸を持たせず、議員別の per-person 軸は記名投票の記録に限定する。発言の個人別集成は引き続き作らない（[[000004]] 不変）。**

- **横断タイムライン**: 1項目 = 1 `observation.event`（状態ノード = イベント型 `ResourceObserved/Changed/Vanished/Restored`）。global `seq` 降順、keyset（seq カーソル）ページング。本文（人間可読な「何が動いたか」）は解釈面の各 projection から注入する。**フィルタは横断・機械的のみ**（`source` / `event_type` / `classification` / 日付範囲 / 構造化キーワード = `title`・`subtitle` への trigram マッチ）。**`person_id`・話者フィルタは持たせない。** 実体化せず**読取時合成**（`observation.event ⨝ 各リードモデル ON observation_seq`、`port.TimelineReader` + `postgres/timeline.go`）で出し、新 projector・新マイグレーション・リードモデル独立性不変条件への抵触を避ける。差分の中身はタイムラインに出さず件数＋分類チップに留め、詳細ビューへリンクする（§7）。
- **議員別記名投票記録**: `ListLegislatorVotes(person_id)` を追加し、ある議員がどう投票したかを**議案 + 結果 + 会議リンクの文脈付き**で返す。**高 `identity_confidence` の同定のみ集成**し、同名異人（低 confidence）の投票は集成しない。`interpretation.vote(person_id)` にインデックスを追加する。
- **非対称の明文化**: `speech.person_id` は**意図的に未索引・per-person 発言ビューなし**（[[000004]]）。`vote.person_id` は**索引・per-person 投票記録あり**（本 ADR）。理由 — 発言は著作物（40条但書）であり集成が doxxing に寄与する、投票は事実記録であり公職者の説明責任の核である。
- **キーワード軸**は構造化フィールド（法令名/番号・議案・会議/委員会名・分類）に限定し、発言本文の全文検索は持たない。スニペットの横断収集＝裏口の発言集成・脱文脈化（§7）・私人名検索による doxxing を防ぐため。
- 検討した代替案と却下理由:
  - **person 軸なし（最保守）**: `vote.person_id` を使わず議員別投票記録も出さない。最も安全だが「自分の代表がどう投票したか」という説明責任の核を提供できず、§11・§12（Popolo 立法トラッカーとの協調）の目的を損なう。→ 却下。
  - **議員の全活動（投票＋発言＋出席）を集成**: §11 の文字通りだが [[000004]]（発言集成・40条但書）と §4 に正面違反。→ 却下。
  - **発言本文の全文検索キーワード**: 強力だが FTS インフラ＋スニペット横断集成（裏口の発言集成）＋私人名検索 doxxing。→ 却下（構造化フィールドに限定）。
  - **タイムラインを実体化（projector が他リードモデルを fold）**: リードモデル独立性不変条件を破り reproject 順序依存を生む。→ 却下（読取時合成）。

本決定は passive / read-only（タイムラインは観測ログの読取 projection）、append-only hash-chain（観測面に書き戻さない、§4-3）、plane 分離（背骨＝観測面・本文＝解釈面、§4-4）、反 doxxing（§5-B / DISCIPLINE §4）、コンテキスト保全（差分は詳細ビューのみ、§7）の設計原則と整合する。

## Consequences

### Pros

- §11 の「議員ウォッチ」を、法（40条但書）と規律（§4）の双方を満たす**唯一の形（記名投票に限定）**で実現できる。
- タイムラインが person 軸を持たないことで、横断名寄せ・プロファイル生成の経路を**構造的に塞ぐ**。
- 読取時合成によりタイムラインは新 projector/migration 不要で、リードモデル独立性不変条件に触れない。`ListTimeline` の proto 契約を安定させれば、将来ソースが増えたとき実体化 headline へ透明に昇格できる。
- speech と vote の非対称を明文化し、将来の実装者が「なぜ vote だけ person 索引があるのか」を辿れる（[[000004]] と本 ADR の単一線として一本化）。

### Cons / Tradeoffs

- 「タイムラインに person 軸を入れない」「キーワードは構造化のみ」を UI / API で恒常的に守る規律コストが生じる（[[000004]] と同種）。
- `identity_confidence` が低い議員（同名異人）の投票は集成されず、その議員の投票記録は不完全/非表示になる（保守側に倒す代償）。
- 読取時合成はソースが増えると join 分岐が増える。性能・ソース数が要求すれば実体化 headline へ昇格する判断が将来要る（proto 契約は不変）。
- 記名投票記録は「公職者の監視は正当」という解釈に依存する。投票者に私人が混ざり得るソース（将来の地方議会等）を足す場合は、本 ADR の前提（投票者 = 全員公職者）を再評価する。
