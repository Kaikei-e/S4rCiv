# ADR-000007: ウォッチ＆アラートをフィード＋端末内保存で実装しサーバ push を採らない

## Status

Accepted

## Date

2026-06-03

## Affected Services

- web (SvelteKit) — **Atom フィード**を `+server.ts` が `ListTimeline` 結果から描画。ウォッチリストは **localStorage**（端末内）。アプリ内ハイライトに使用
- api (Go) — **追加 RPC なし**（`ListTimeline` を再利用）。サーバはユーザ状態・購読ストア・外向き通知器を一切持たない
- （持たないものの明示）アカウント、サーバ側購読ストア、外向き通知器（apprise / メール / Webhook）

## Context

概念 §11 は M3 を「タイムライン＋議員/キーワードのウォッチ＆アラート（**apprise 的通知**）」と定義する。一方で概念 §14.7 は「**アラートの配信境界（opt-in、レート、個人を狙い撃たない設計の具体）**」を明示的な**未決事項**として残している。

設計原則と規律はこの領域に強い制約を課す:

- §4-1 **受動・読取専用**: 公開エンドポイントへの HTTP GET のみ。**書き込み・自動アクションをしない。** S4rCiv は "sentinel"（見張り）であって行為者ではない。
- **no-auth・単一自己ホストバイナリ**: 認証もユーザアカウントも前提にしない。
- DISCIPLINE §8 **通知/アラート**: auto-post で個人を狙い撃たない（DiffEngine 的運用を採らない）。opt-in 以外で配信しない。
- DISCIPLINE §7 / §9: スニペット単体を結論にしない、出典・取得時刻を常に併記する。

「ウォッチリスト」「通知先」を**サーバに持つ**と、identity と購読の保存（= 書き込み）＋宛先への外向き送信（= 自動アクション）が必要になり、§4-1・no-auth・§8 と緊張する。M3 でこの配信境界（§14.7）を確定する必要があった。

## Decision

**ウォッチ＆アラートを「サーバが発行する標準フィード（Atom）＋ブラウザ端末内（localStorage）のウォッチリスト」で実装する。サーバ側のアカウント・購読ストア・外向き通知器（apprise / メール / Webhook）は持たない。S4rCiv はフィードを発行するだけで、個人に push しない。**

- **フィード発行**: 任意のタイムラインフィルタ（`source` / `event_type` / `classification` / 日付 / 構造化キーワード）・法令・委員会・議員記名投票記録に対し、購読可能な **Atom フィード URL** を発行する。フィルタは URL に符号化する。フィードは SvelteKit の `+server.ts` が `ListTimeline`（[[000006]]）の結果を Atom 描画する（**新 RPC・新ストア不要**）。
- **購読と通知はユーザ側**: ポーリング・通知はユーザ自身のフィードリーダー / apprise が担う。S4rCiv は発行するのみで、個人に push しない。これにより **opt-in が構造的に保証**され（ユーザが自分のリーダーに足す行為そのものが同意）、**auto-post による個人攻撃の経路が存在しない**（§8）。
- **アプリ内ウォッチリスト**: localStorage に保持し、来訪時のハイライトに使う（**サーバ状態なし**・端末ローカル）。
- **全フィードエントリ**に出典 permalink ＋取得時刻＋分類（自動/未レビューを明示）を含め、スニペット単体で結論にしない（§7・§9）。
- **§14.7 の確定**: アラートの配信境界を「**S4rCiv は配信せず発行する**」と定める。レートは読取専用 API 側のソース別レート制御で守り、宛先管理・狙い撃ち防止は「宛先を持たない」ことで構造的に達成する。
- 検討した代替案と却下理由:
  - **サーバ購読＋apprise 配信**（§11 の文字通り）: identity / 購読の保存（書き込み）＋外向き通知器が必要で §4-1・no-auth と緊張する。配信レート・宛先管理・狙い撃ち防止の運用負担を負う。→ v0 では却下（自己ホスト運用者が opt-in で足す拡張余地は残す）。
  - **アプリ内ハイライトのみ（フィードなし）**: 最小だが「アラート」が「来訪時に見える」だけになり §11 の通知を満たさない。→ 却下（フィードで外部リーダー通知を可能にする）。

本決定は passive / read-only（§4-1）、標準準拠でサイロを作らない（§5、Atom）、opt-in・個人を狙い撃たない通知（§8）、出典・コンテキスト保全（§7 / §9）の設計原則と整合する。

## Consequences

### Pros

- §4-1（受動・行為者でない）・no-auth・単一バイナリ・§8（opt-in・個人を狙い撃たない）を**構造的に**満たす。「宛先を持たない」ことで狙い撃ちが原理的に不可能になる。
- 標準準拠（Atom）でサイロを作らない（§5）。既存のフィードリーダー / apprise 資産にそのまま乗る。
- サーバにユーザ identity を持たないため、プライバシー面の攻撃面が小さい。
- 新 RPC・新ストア不要（`ListTimeline` 再利用）。実装が薄い。

### Cons / Tradeoffs

- 「リッチな push 通知」体験は提供しない（ユーザがフィードリーダー / apprise を用意する手間が要る）。
- ウォッチリストが端末ローカルなので**端末間で同期されない**（共有はフィード URL のコピーで代替）。
- フィードのポーリング頻度はユーザのリーダー任せで、S4rCiv 側でレート保証できない（ただし読取専用 API のソース別レート制御で過負荷は防ぐ）。
- 将来サーバ push を足す場合、identity・配信境界（§14.7）・狙い撃ち防止の設計を本 ADR の前提に追加で起こす必要がある。
