# ADR-000024: egov-law 差分 read model の空 node_changes を [] で直列化し timeline を scalar 耐性化・reproject を読者原子的にする

## Status

Accepted

## Date

2026-06-09

## Affected Services

- `egov-differ`（Go, `internal/usecase/diff`） — 構造差分が空のとき `node_changes` を JSON `null` ではなく `[]` で直列化（nil スライス対策）
- `api` query 側（Go, `internal/driver/postgres/timeline.go`） — `ListTimeline` の差分カウントを LATERAL + `count(*) FILTER` に集約し、非配列 `node_changes` を空配列扱いするガードを追加
- `interpretation.change`（migration `20260603000017`） — `UNIQUE(observation_seq)` 追加、`ApplyChange` を upsert 化、`BeginRebuild` を truncate 廃止（読者原子的 reproject）

## Context

公開タイムラインが「⊘ 取得に失敗しました」だけを返し全断した（[postmortem 2026-06-09](../postmortem/2026-06-09-timeline-null-node-changes-22023-outage.md)）。原因は `ListTimeline`（横断タイムラインを read 時に合成する、[[000006]]）が `interpretation.change` の差分件数を

```sql
jsonb_array_elements(c.diff->'node_changes')
```

で展開していたところに、`node_changes` が JSON `null`（スカラ）の行が 1 件混ざっていたこと。Postgres は scalar に対し `jsonb_array_elements` を呼ぶと `SQLSTATE 22023`（`cannot extract elements from a scalar`）を上げる。これが SELECT リスト内の式だったため、**たった 1 行の不正データがクエリ全体を失敗させ**、タイムライン全体が落ちた。

`node_changes = null` の出所は egov-differ（[[000005]] で differ は Connect-RPC、永続化は Go 側）。差分を `interpretation.change.diff` に焼くとき、構造差分が 0 件（administrative な内容変更で article/clause レベルの変化なし）だと Go のスライスが `nil` のままになり、`json.Marshal(nil slice)` が `[]` ではなく `null` を出していた。`interpretation.change` は [[000002]] の解釈面 Tier-2（捨てて再計算できる disposable read model）なので、観測面の ground truth は無傷で、修復は projector の再計算（reproject）で完結できる。

加えて監査（immutable-design-guard）で、`interpretation.change` が `observation_seq` に UNIQUE を持たず（PK は合成 bigserial）、`ApplyChange` が素の INSERT、`BeginRebuild` だけが delete→replay で重複を防いでいたことが分かった。これは [[000022]] で kokkai 側 projector が TRUNCATE 廃止の upsert 上書き（読者原子的 reproject）へ移行した方針と不一致で、egov 差分 projector だけが reproject 中に enrichment の消える窓を残していた。

## Decision

次の 3 点を一体で採用する。

### 決定1: egov-differ は `node_changes` を常に配列で直列化する

`diffJSON.NodeChanges` を nil のままにせず、空でも `[]nodeChangeJSON{}` で初期化する。これで構造差分 0 件の administrative な変更も `"node_changes":[]` になり、scalar 化しない。これが根本原因の修正であり、`差分が空 → [] で直列化` を回帰テストで固定する。

### 決定2: `ListTimeline` を非配列 `node_changes` に耐性化する

3 本の相関サブクエリ（added/deleted/modified それぞれで `jsonb_array_elements`）を 1 本の `LEFT JOIN LATERAL` に集約し、`count(*) FILTER (WHERE op=...)` でカウントを導出する。配列ガードは 1 箇所の `CASE` に集約する。

```sql
jsonb_array_elements(
  CASE WHEN jsonb_typeof(c.diff->'node_changes') = 'array'
       THEN c.diff->'node_changes' ELSE '[]'::jsonb END)
```

決定1 で今後 `null` は生まれないが、**read model の 1 行の不正がタイムライン全体を落とせる構造そのものを消す**ための多重防御。`null` は「差分 0 件」と同義に degrade する（嘘の値を作らない）。展開も 3 回から 1 回に減る。

### 決定3: `interpretation.change` を UNIQUE + upsert + truncate-less reproject にする

migration `20260603000017` で `UNIQUE(observation_seq)` を追加し（既存の非 UNIQUE index は冗長になるので削除）、`ApplyChange` を `ON CONFLICT (observation_seq) DO UPDATE SET = EXCLUDED.*` の upsert にする。差分は 1 つの `ResourceChanged` から決定的に再計算される全量上書きなので、SQL に business CASE も COALESCE 保持も持ち込まない（merge-safe upsert）。これにより `BeginRebuild` から delete を外し（offset/rebuilding リセットのみ）、reproject 中も各行をその場上書きして enrichment を消さない（[[000022]] と整合）。二重 apply（reproject と常駐 daemon の競合等）は UNIQUE 違反で fail-loud になり、タイムラインに同一行が二重表示される事故を防ぐ。明示全消去が要る場合のため `TruncateEgov` は残す。

### 検討して却下した代替案

- **read model の該当 1 行を手書き SQL で `[]` に UPDATE するだけ**: 最速で全断は止まるが、disposable read model を観測ログ経由せず直接 mutate することになり、根本原因（producer の nil→null）も残る。決定1 + reproject の方が原則（[[000002]]）に沿う。
- **決定2（consumer ガード）を入れず producer 修正のみ**: 既存の毒行が無害化されず、また「1 行で全断」の脆さが残る。可用性のため多重防御を採る。
- **UNIQUE を入れず delete→replay のまま**: reproject-safe ではあるが [[000022]] と不一致で、reproject 中の enrichment 消失窓が残る。一体で揃える。

## Consequences

### Pros

- 構造差分 0 件の egov-law 変更が `node_changes:[]` になり、scalar 22023 が出ない（根本修正）。
- read model の 1 行が壊れてもタイムライン全体は落ちず、その行が「差分 0 件」に degrade するだけになる（多重防御）。
- egov 差分 reproject が読者原子的になり、reproject 中もタイムラインの差分カウントが消えない（[[000022]] と方針統一）。
- `observation_seq` の一意性が DB 不変条件になり、二重 apply が静かな重複でなく即エラーになる。
- 差分カウントの展開が 3 回 → 1 回に減り、SQL の重複も解消する。

### Cons / Tradeoffs

- consumer 側ガードは不正データを「差分 0 件」へ静かに丸める。producer 修正後は発生しない想定だが、将来別経路で `node_changes` が壊れても気付きにくい（postmortem に再発時の確認手順を残す）。
- `interpretation.change` への書き込みが INSERT から upsert になり、`ON CONFLICT` 句のぶん SQL が増える（at M1 規模では無視できる）。
- reproject は law read model も同時に再計算する（`egov-law reproject` は両方を回す）。law projector 側の原子性は本 ADR の対象外で、別途の課題として残る。
