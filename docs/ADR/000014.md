# ADR-000014: 完全性検証を per-record バッジから外し事案ページの端末内有界検証＋署名・外部アンカー付きチェックポイントとして設計する

## Status

Accepted

## Date

2026-06-05

## Affected Services

- web (SvelteKit) — Provenance chip 再設計（「(未検証)」撤去・「記録 #seq」を事案ページの検証パネルへ deep-link）。事案ページ（議事録・法令）に検証集約パネル＋**端末内 WASM/JS verifier** を新設
- api (Go) — **検証用の新 read 面**: ある stream（事案）の全 event について `HashableEvent` 全フィールド＋`log_hash`/`log_prev_hash`/`content_hash`＋被覆チェックポイント（署名・外部アンカー参照付き）を返す読取専用 RPC。表示用に切り詰めた `Attribution`（[[000006]]）とは別 shape
- observation (db) / collector — `observation.checkpoint`（[[000001]] / [[000003]]）に署名・外部アンカー参照を持たせ、定期的に外部へ push する新ジョブ

## Context

Timeline の各項目に出る Provenance chip が、**全件で無条件に「(未検証)」を表示していた**（`ProvenanceChip.svelte` の静的ラベル、`{#if}` で分岐しない）。これは設計上 2 つの異なる概念を取り違えていた:

- **連結（chain linkage）**: その記録が append-only ハッシュ連鎖の中に座っている事実（`seq`・`log_hash`・`log_prev_hash` を持つ）。chip が実際に出していたのはこれ。
- **完全性検証（integrity verification）**: ハッシュを正規形（`HashableEvent`、[[000003]]）から再計算し、`seq` 順に連鎖の連続を確かめてログ改ざんが無いことを示す行為。

chip は連結を表示しているだけで検証はしておらず、検証する側のコードもフィールドも未実装だったため、**全件が等しく未検証**になっていた。バグで `false` に張り付いていたのではなく、検証機構そのものが無かった。

さらに概念上の不整合があった。tamper-evidence は CORE_CONCEPT §5 の脅威表のとおり **global log chain ＋ 定期署名チェックポイント**で担保される、本質的に**連鎖単位**の性質である。にもかかわらず chip は**レコード単位**で検証状態を出そうとしていた — これはカテゴリーエラーで、1 レコードの `log_hash` 単体は連鎖の連続性なしには検証対象にならない。

加えて、chip が露出するのは 8 文字に切り詰めた `log_hash` だけで、`HashableEvent` の各フィールド値を第三者に渡す export も存在しない。結果として CORE_CONCEPT §13「観測面のハッシュ連鎖により、誰でも記録の完全性を検証可能」は、実装上は**誰にも不可能**なアスピレーションに留まっていた。§5 が想定する検証者は「S4rCiv 自身のログ改ざん疑義」を晴らしたい**敵対的な第三者**であり、S4rCiv が自分の記録に自分で緑の「検証済み」を付けても当の相手は信じない。

本決定はこの取り違えを正し、「未検証」表示を意味のある検証アーキテクチャに置き換えるために必要だった。設計原則との関係: passive / read-only（検証は GET 由来の観測ログに対する再計算のみ）、append-only hash-chain（§4-3）、plane 分離（検証は観測面に閉じる）、source compliance（出典・取得時刻の常時併記、DISCIPLINE §7/§9）と整合する。

## Decision

**完全性検証を per-record の検証バッジから外し、(1) chip は provenance に徹し、(2) 検証は事案ページの端末内（ブラウザ）有界検証として実装し、(3) その信頼の基準点を署名済み＋外部アンカー付きのチェックポイントに置く。ハッシュ正規形は [[000003]] の `proto-linked-v1` を維持する（本 ADR は [[000003]] を supersede しない）。**

1. **検証の単位を連鎖／チェックポイント単位に画定する。** レコード単位の検証バッジを廃止する。完全性は「ある記録が検証済みか」ではなく「連鎖が直近チェックポイントまで連続しているか」で語る。

2. **per-record chip は provenance に徹する。** chip は **出典＋最終取得＋「記録 #seq ↗」**のみを出し、切り詰めハッシュ片と「(未検証)」を撤去する。`#seq` は事案ページの検証パネル（当該レコードにアンカー）への deep-link であり、`seq` は安定した引用ハンドルとしても機能する。chip の本体価値は DISCIPLINE §7/§9 が要求する出典 permalink ＋取得時刻にある。

3. **事案ページ（Resource 単位＝議事録・法令）に検証集約パネルを置き、端末内で有界検証する。** ブラウザが次を再計算する: ①その事案の各記録の `log_hash` を正規形 `HashableEvent` から再計算 ②その事案の content chain（stream 内の `prev_content_hash` 連続） ③それらが直近の**署名済みチェックポイント**の範囲に収まること。global log chain（全 Stream を挿入順に貫く単一連鎖）を genesis から毎回ブラウザで通し再計算するのは非現実的（将来数百万 event）なので**有界化**し、全件の通し再計算は export を落とした第三者ミラーに委ねる。これにより「ユーザーごとに簡単に・任意の言語で検証できる」（§13）を、自己申告に頼らず**読者の実機上**で満たす。なお chip も検証パネルも**市民向けの平易な言語**で提示し、ADR 番号・checkpoint・hash・proto 等の内部用語を一次ラベルとして表に出さない（内部用語はコード／ドキュメントと専門者向けの副次表示に留める）。

4. **チェックポイントの信頼を署名＋外部アンカーで担保する。** `observation.checkpoint` を S4rCiv 鍵（ed25519 等）で署名し、定期的に Internet Archive (Memento) / OpenTimestamps 等の第三者へ push して独立したタイムスタンプとコピーを残す。S4rCiv が鍵で過去ごと再署名し直しても、外部に残った当時の値と食い違うため検知できる。これにより §5「自ログ改ざん疑義」を構造的に閉じる。

5. **ハッシュ正規形は `proto-linked-v1` を維持し、in-browser verifier を CDC テストで同梱する。** ブラウザの verifier は Go の `proto.MarshalOptions{Deterministic: true}` とバイト一致を CDC（consumer-driven contract）テストで保証して同梱（WASM/JS）する。`HashableEvent` が scalar-only / map 無しゆえ field-number 順エンコードが実装間で実質安定する（[[000003]]）前提に乗る。

検討した代替案と却下理由:

- **per-record のサーバ検証結果（緑の「検証済み」）を出す**: UI は最軽量だが、§5 の当の敵対者（S4rCiv 不信）は S4rCiv 自身の✓を信じない。「自分の答案を自分で採点」になり検証の意味を成さない。→ 却下。
- **チェックポイントを S4rCiv 署名のみ（外部アンカー無し）にする**: 鍵漏洩が無い限り「第三者がこっそり差し替えた」は防げるが、鍵を持つ S4rCiv 自身の全面再署名を外部から検知できない。→ 却下し、外部アンカーを必須とする。
- **透明性ログ＋複数 witness の co-sign（Certificate Transparency 流）**: 最も強いが witness 運用・ガバナンスの負担が重く v0 には過剰。→ 却下（将来拡張余地は残す）。
- **ハッシュ正規形を JCS（RFC 8785）へ今切替える**: 「全ユーザーが任意言語で検証」という新要件は、[[000003]] が「将来 verifier の移植性要件が強まれば再評価」と明記した再評価条件をまさに発火させる。JCS なら別言語 verifier は「JSON 正規化＋sha256」だけで済む。しかし `proto-linked-v1` との 2 形式併存（M1 event は既に `proto-linked-v1`）が永続し、[[000003]] が避けたかった hash 専用形式の二重管理を背負う。→ 今は不採用。`alg_version` で将来の切替余地は開けたまま残す。
- **chip から chain 参照を全撤去する**: 最もすっきりするが、per-row の「この記録を検証」導線を失う。→ 最小アンカー（記録 #seq）＋ deep-link を採る。

本決定は passive / read-only（§4-1）、append-only hash-chain（§4-3）、標準準拠で第三者検証可能（§13・§5）、出典・コンテキスト保全（§7 / §9）の設計原則と整合する。

## Consequences

### Pros

- 「(未検証)」が運んでいた誤った含意（市民に「このデータは信用できない」と読ませる不安語）を除去し、chip を本来の provenance（出典＋取得時刻）に戻す。検証の単位が連鎖／チェックポイントに揃い、カテゴリーエラーが解消する。
- §13「誰でも独立に検証可能」が**実装上も真**になる。検証が読者の実機（ブラウザ）で走るため、S4rCiv の自己申告に依存しない。§5 の敵対的第三者という想定にも噛み合う。
- 署名＋外部アンカー（Internet Archive / OpenTimestamps）により、S4rCiv 自身の全面再署名でも検知可能になり、§5「自ログ改ざん疑義」を構造的に閉じる。§5 が挙げる Internet Archive 経由の第三者証跡化とも整合する。
- ハッシュ正規形を据え置くため、既存 M1 event（`proto-linked-v1`）の再ハッシュや移行が不要。決定の不可逆部分（log_hash）に手を触れない。
- 有界検証により、巨大化する global chain でもブラウザ内検証が一定コストに収まる。全件再計算は export を介して第三者ミラーに委ねられる。

### Cons / Tradeoffs

- 別言語の独立 verifier は `HashableEvent` の protobuf Deterministic marshal を再実装する必要が残る（[[000003]] が認めた「移植のハードルがわずかに高い」点）。JCS を採れば消えたコストだが、二重管理を避けるため据え置いた。`alg_version` による将来の JCS 再評価は積み残し。
- in-browser verifier は Go の marshal とバイト一致であり続けねばならず、proto ライブラリ更新や `HashableEvent` への map/optional 混入で連鎖検証が静かに壊れる（immutable-design-guard の監査対象）。CDC テストの維持が必須。
- 検証用の新 read 面（per-stream export）と、チェックポイント署名＋外部 push ジョブという新規実装が必要。まず per-stream export を出し、全件 dump（seq 範囲ページング）は後追いとするため、当面は事案単位の有界検証に限られる。
- 外部アンカー先（Internet Archive / OpenTimestamps）の可用性・仕様変更に依存する運用面が増える。push 失敗時の扱い（再送・欠損記録）は別途運用設計が要る。
- チェックポイント署名鍵の管理（生成・保管・ローテーション）という運用責務が新たに生じる。鍵が漏れれば自己署名部分の保証は崩れる（外部アンカーが最終的な保険として残る）。

## Implementation Notes (v0)

実装時に確定した事項（Web 調査を踏まえた解像度上げ。決定本体は上の Decision を変えない）。

1. **「WASM/JS」を JS/TS に画定**: 端末内 verifier は **TypeScript（protobuf-es `toBinary` ＋ WebCrypto `crypto.subtle.digest('SHA-256')`）**で実装する。決め手は (a) protobuf の Deterministic marshal は公式に canonical／移植安定を保証せず（protobuf.dev「serialization-not-canonical」）、byte 一致の担保は**どの実装形態でも CDC golden-vector テスト**に拠る ⇒「同じ Go コードを WASM 化すれば安心」という WASM 最大の動機が実質消える、(b) 標準 Go→WASM は ~2MB（brotli ~500KB）＋ `wasm_exec.js`（Go 版と結合）でスマホ前提の civic サイトには重い、(c) TinyGo は `google.golang.org/protobuf`（reflection 多用）を動かせず別 marshaler になり「同一コード」利点も消える、(d) scalar-only の `HashableEvent` では protobuf-es が Go と既に byte 一致（vendored ライブラリ実装で確認）。`alg_version`（[[000003]]）で将来 WASM/JCS への退路は維持。

2. **v0 は外部 push を行わない**: 署名（ed25519）＋外部アンカー（Internet Archive / OpenTimestamps）push は **後続マイルストーンに延期**（鍵管理・outbound network・push 失敗時の運用設計が要り、1 サイクルで検証完了できないため）。代わりに **CDC ＋ verifier テストを徹底**して byte 一致の不可逆部分を先に固める。検証パネルは「チェックポイントは S4rCiv の自己コミットメントで外部アンカー未了」「全件の通し再計算は公開エクスポート経由で第三者ミラーに委ねる」を平易な言葉で明示し、緑の「検証済み」を S4rCiv が名乗らない（§5 のカテゴリーエラーを別の場所で再発させない）。

3. **新 read 面 = `GetStreamVerification(stream_id)`**: ある Stream の各 event の canonical `HashableEvent`（[[000003]] の正規形＝ハッシュ対象そのもの）＋格納 `log_hash`＋被覆チェックポイント（v0 は未署名・しばしば未生成）を返す。Go 側は格納行を `EventFacts` に戻し**同一の `Hashable()` 射影**で canonical を再構成する（SQL で canonical を別途組み立てない）。`Attribution` に `stream_id` を追加し、UI が prefix を端末側で再構成せず記録 #seq を検証パネルへ deep-link できるようにした。

4. **端末内検証は「連鎖単位の有界」の中でもさらに stream 局所**: log chain は global（全 Stream が `seq` で交錯）ゆえ、単一 Stream の export では global な `log_prev_hash` 連続は確かめられない。v0 パネルが行うのは §3 のうち ①各 `log_hash` の再計算一致 と ②その Stream の content chain（`prev_content_hash` 連続）であり、全件の `seq` 順連続は第三者の全ミラー（export）に委ねる。これは [CONTEXT.md](../../CONTEXT.md) の「完全性検証」（連鎖全体の性質）の**有界な部分実施**であって矛盾しない。

5. **CDC 契約**: Go 側（`services/api/internal/domain/observation/golden_test.go`）が golden vector（`testdata/hashable_golden.json`：genesis 64-zero・空 optional・`stream_seq` > 2^53・zero/UNSPECIFIED 省略・4 event 連鎖）を産出し、TS verifier の CDC テスト（`web/src/lib/verification/verifier.cdc.test.ts`）が wire バイトと `log_hash` を 1 つ残らず再現する。`HashableEvent` への map/optional 混入や proto ライブラリ更新で byte 一致が崩れれば即 fail（scalar-only 不変条件と immutable-design-guard の監査対象）。
