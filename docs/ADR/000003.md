# ADR-000003: 観測面イベントのハッシュ正規形に protobuf HashableEvent を採用し Connect-RPC をサービス境界に導入する

## Status

Accepted

## Date

2026-06-03

## Affected Services

- api (Go) — Connect-RPC の読取専用 query サーバ、`HashableEvent` による `log_hash` 検証
- collector (Go) — observation event の append 時に `HashableEvent` から `log_hash` を計算（新設プロセス）
- differ (Rust) — 将来 `interpretation` 投影で同一 proto スキーマを共有
- db (Postgres) — `observation.checkpoint.alg_version` に `'proto-linked-v1'` を採用

## Context

[[000001]] は `log_hash` の計算を**アプリ所有**と決めたが、その canonical 化仕様（どの列を・
どんな正規形で連結して sha256 するか）は未確定のまま残した（ADR-000001 Cons「ハッシュ対象列の
canonical 化仕様をアプリと verifier で version 同期し続ける規律が要る」）。

観測面の core value は **誰でも独立に記録の完全性を検証できる**こと（CORE_CONCEPT §13 再現性、
§4-3 tamper-evidence）。したがって正規形は「第三者が任意の言語で verifier を実装でき、同じ
バイト列に再計算できる」ことを満たす必要がある。M1（国会会議録 adapter）で最初の event を
append する瞬間にこの正規形が要る。

あわせて、collector（command 側）/ api（query 側）/ 将来の differ というサービス境界をどの RPC
層で型付けするかも未決だった。スキーマ駆動で型を共有できれば、hash 対象の型定義と RPC の型定義を
単一ソースに集約できる。

## Decision

**観測面イベントのハッシュ正規形に scalar-only の protobuf message `HashableEvent` を採用し、
`log_hash = sha256(Deterministic marshal(HashableEvent))` とする。あわせてサービス境界に
Connect-RPC を導入する。**

- `HashableEvent` は `log_hash` の対象列だけを持つ: `event_id` / `stream_id` / `stream_seq` /
  `type` / `source` / `fetcher_version` / `observed_at` / `source_published_at` / `content_hash` /
  `prev_content_hash` / `log_prev_hash`。**map / optional / 任意フィールドを置かず、全フィールドを
  明示 populate** する。
- marshal は `proto.MarshalOptions{Deterministic: true}`。scalar only・map 無しなら field-number
  順のエンコードが実装間で実質安定する。
- 正規形 version を `observation.checkpoint.alg_version='proto-linked-v1'` に固定する。スキーマ
  変更は version bump（`'proto-linked-v2'` 等）で表現し、過去 event は当時の version で検証する。
- サービス境界に **Connect-RPC**（protobuf スキーマ駆動）を採用する。query api は read model を
  返す**読取専用** service とし、`HashableEvent` はその proto 定義群の安定サブセットとして共有する。
- 検討した代替案と却下理由:
  - **JSON Canonical（RFC 8785 JCS）+ sha256**: 言語非依存の移植性は最高。しかし protobuf を
    schema / RPC に使うなら hash だけ別形式にするのは二重管理になる。→ 却下（将来 verifier の
    移植性要件が強まれば再評価する余地は残す）。
  - **Connect-RPC の envelope メッセージをそのまま marshal**: 単純だが、envelope に将来 map や
    任意フィールドが入ると canonical 性が静かに壊れる。→ 却下し、hash 専用の安定サブセット型
    `HashableEvent` を分離する。
  - **protobuf wire をそのまま canonical とみなす**: protobuf serialization は仕様上 canonical を
    保証しない（map キー順・未知フィールド・default/zero 値の省略規則が実装差を生む）。→
    scalar-only / map 禁止 / 全 populate という制約で実質安定化したうえで採用する。

本決定は passive / read-only（hash 対象は GET 由来の観測のみ）、append-only hash-chain（§4-3）、
plane 分離（hash は観測面のみ）の設計原則と矛盾しない。

## Consequences

### Pros

- 第三者検証が任意言語で可能になる。proto スキーマと sha256 さえ実装すれば `log_hash` を再計算
  でき、§13 の再現性を具体化する。
- schema / RPC / hash 対象が単一の proto 定義に集約され、collector / api / 将来 differ で型を共有
  できる。
- `alg_version` により hash scheme を schema 変更なしに進化させられる（[[000001]] の `checkpoint`
  機構にそのまま相乗りする）。

### Cons / Tradeoffs

- protobuf wire の非 canonical 性を「scalar-only / map 禁止 / 全 populate」という規律で回避して
  いる。`HashableEvent` にうっかり map や optional を足すと連鎖検証が静かに壊れる（
  immutable-design-guard の監査対象そのもの）。
- Deterministic marshal の安定性は同一スキーマ前提であり、proto ライブラリの major 変更時は
  `alg_version` bump と再検証の規律が要る。
- `buf` / `connect-go` / `protobuf-go` という新規ツールチェーン依存が入る（go.mod は従来ゼロ依存）。
- verifier 実装が proto 定義に依存するため、JCS のような完全に言語非依存な形式よりは移植の
  ハードルがわずかに高い。
