# ADR-000011: e-Gov 法令の存在をメタデータで確認し本文ファイル404を消失と扱わない

## Status

Accepted

## Date

2026-06-05

## Affected Services

- collector (Go) — `internal/gateway/egov`（`Fetch` の404/空本文処理、`lawExists` 存在オラクル）
- collector (Go) — `internal/usecase/collect`（`FetchResult.ContentUnavailable` のスキップと `MarkPending` 再ポーリング）
- port (Go) — `FetchResult.ContentUnavailable`、`ControlStore.MarkPending`
- db (Postgres) — `control.poll_state`（失敗カウントを増やさない短期再ポーリング）

## Context

肝入りの差分機能（e-Gov 法令の現行本文を Stream として観測し、改正を `ResourceChanged` 列として記録する）で、現に有効な法令が誤って `ResourceVanished`（消失）として不変ハッシュ連鎖ログに記録される事象を発見した。

具体例（2026-06-05 にライブ API で再現確認）:

- 411AC0000000136「組織的な犯罪の処罰及び犯罪収益の規制等に関する法律」(平成11年法律第136号) は、2026-06-04 に正常観測（`ResourceObserved`、722 ノード）された翌日、`ResourceVanished` として記録された。
- しかし当該法令は現に有効。`/laws?law_id=…` メタデータは `repeal_status: None` / `current_revision_status: CurrentEnforced` を返し続けている。
- 原因は **e-Gov の公開順序**: 2026-06-05 施行の改正（令和8年法律第30号）でメタデータの現行リビジョンポインタが新リビジョン `411AC0000000136_20260605_508AC0000000030` に切り替わった一方、**その新リビジョンの法令標準XML本文ファイルがまだ未公開**で、`law_data/{law_id}?law_full_text_format=xml`（およびリビジョンID直指定）が `404004`「指定のパラメータで取得できる法令本文ファイルは存在しません」を返す。旧リビジョンの本文は今も 200 で取得できる。
- 旧実装は `law_data` のあらゆる 404 を `FetchResult{Present:false}` とし、`Decide()` が `ResourceVanished` を発火していた。
- 同日に同じパターンで 6 件の法令（政令含む）が誤消失しており、改正施行日ごとに再発する構造的バグだった。

これは設計原則に直接抵触する: 観測面は append-only・ハッシュ連鎖の ground truth であり（CORE_CONCEPT §4-3 / §9.1）、DISCIPLINE §4-3「観測面のログを改変・捏造しない」は、起きていない消失を書き込まないことも含む。`law_data` の本文ファイル 404 は「本文成果物が取得不能」を意味するのであって、「Resource（レジストリ上の法令）が消えた」を意味しない。両者を取り違えていた。

## Decision

**e-Gov 法令アダプタにおいて、`ResourceVanished` は Resource がソースの権威的な存在シグナル（`/laws` 法令一覧メタデータ）から消えたときにのみ発火させる。本文ファイルの 404／空本文は、メタデータ上に法令が存在する限り消失とせず、`FetchResult.ContentUnavailable`（本文未取得）として観測イベントを生まず短期に再ポーリングする。**

具体には:

1. **存在と内容の分離**: 存在 = `/laws?law_id=X` の `total_count`、観測内容 = 現行リビジョンの本文XMLスナップショット。`FetchResult` に `ContentUnavailable bool` を追加（`Present` と排他）。
2. **存在オラクル**: `law_data` が 404 または空本文を返したら `/laws?law_id=X` を引く。`total_count>0` なら `ContentUnavailable`、`total_count==0` なら `Present:false`（真の消失）。`/laws` 自体が非200なら**確認不能としてエラー**を返し、消失を捏造しない（消失には積極的確証を要求する）。
3. **短期再ポーリング**: `ContentUnavailable` は観測イベントを生まず、`ControlStore.MarkPending` で `next_due_at` を約3時間後に設定（`pollCadence` 24h より短い）。`consecutive_failures` は増やさない（公開ラグは失敗ではない）。新本文が公開され次第、次の観測で自然に `ResourceChanged` として拾われる。
4. **既存の偽消失（global seq 2132 ほか6件）は不変ログを一切編集しない**: append-only を厳守し、新リビジョン本文が公開された時点で `ResourceRestored` が自然発火することで自己回復させる。タンパー証拠の同一性を最優先する。

### 検討した代替案

- **エラーコードのみで判別**（404004 を本文未公開とみなしスキップ）: 追加リクエスト不要だが、存在しない LawId も 404004 を返すため**真の消失を検出できない**。却下。
- **現状維持＋ResourceRestored 自己修復のみ**: 最安だが、改正施行日ごとに偽の「消失→復活」ペアが ground truth に永久に焼かれる。却下。
- **解釈面（projector/timeline）で消失を抑制**: 観測ログは保持できるが、タイムラインの「1 項目＝1 観測イベント」不変（CONTEXT.md）に retroactive な再解釈の例外を持ち込み脆い。却下。

## Consequences

### Pros

- 公開ラグ起因の偽 `ResourceVanished` が不変ログに書かれなくなり、観測面の ground truth が忠実になる（DISCIPLINE §4-3 整合）。
- 改正施行当日でも、新本文の公開後**数時間以内**に `ResourceChanged` として捕捉できる（市民の関心が高いタイミングでの迅速性）。
- 「存在＝権威的メタデータ／内容＝スナップショット」という分離は、将来の他ソース（成果物の取得性と Resource の存在が別シグナルになるもの）にも適用できる一般原則になる。
- 消失の判定が単一成果物の可用性ではなくレジストリ上の存在に根拠を持つため、誤検出に対して堅牢。

### Cons / Tradeoffs

- 本文 404／空本文のたびに `/laws` への追加 GET が1回発生する（レアパスのみ。レート制限により law_data と合わせ約60秒）。
- 既発の偽消失6件は append-only 厳守のため当面ログに残り、新本文公開まで（数時間〜1日以上）デモのタイムラインに「消失」が見える。自己回復後も履歴には `Observed→Vanished(偽)→Restored` が残る。
- `MarkPending` という再ポーリング経路が増え、`control.poll_state` の状態遷移がやや複雑になる。
- `/laws` を唯一の存在オラクルに依存する（e-Gov がメタデータ側を先に削除し本文を残すような逆順公開をした場合の挙動は未検証）。
