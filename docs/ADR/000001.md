# ADR-000001: 観測面を append-only ハッシュ連鎖ログとする三面 DB スキーマを採用する

## Status

Accepted

## Date

2026-06-03

## Affected Services

- db (Postgres 18 / Atlas) — `observation` / `interpretation` / `control` の三 schema を新設（`db/migrations/2026060300000{1,2,3}_*.sql`）
- api (Go) — `observation` の読取と append、ハッシュ連鎖の検証
- differ (Rust) — snapshot を取得し diff を `interpretation` へ投影

## Context

S4rCiv の core value は「いつ・何が・どう変わったか／消されたか」を辿れること、そして
**S4rCiv 自身が歴史を書き換えていない証跡（tamper-evidence）**を第三者へ提供することにある
（CORE_CONCEPT §1, §4-3）。ストレージはスキャフォールド時点で Postgres 18 + Atlas に確定して
いる（概念 §8 の「組み込み・ゼロ依存」理想とは乖離するが、実装はこちらが正）。

DB に載るデータには性質の異なる 3 種が混在する:

1. 改ざん耐性を要する不変の ground truth（生スナップショットと変化イベント）
2. そこから再計算可能な projection（正規化・要約・分類）
3. どちらでもない可変の運用状態（polling cursor・rate-limit・robots・watch-list）

これを 1 つの面に混ぜると、不変性（1）か再現性（2）のどちらかが必ず壊れる。収集は
DISCIPLINE §1 により serial + interval で行うため書込量は穏当で、global な直列化を許せる。

## Decision

DB を **`observation`（不変）/ `interpretation`（projection）/ `control`（可変運用）の三 schema に
物理分離**し、観測面を次の構造で **append-only ハッシュ連鎖イベントログ**とする。

- **二重のハッシュ連鎖**: stream 別の content 連鎖（`prev_content_hash`）と、全 stream を挿入順に
  貫く **global 単一 log 連鎖**（`log_prev_hash` / `log_hash`）。
  - 代替案 **stream 別の log 連鎖** は却下: 「ある stream の末尾ごと削除」を外部 index 無しに
    検出できず、台帳全体の主張が弱い。
  - 代替案 **最初から Merkle のみ** は却下: M0 にオーバーヘッド。後付け可能（下記 checkpoint）。
- **連鎖の責務はハイブリッド**: DB が単一行 `chain_head` を `FOR UPDATE` で直列化し、
  `log_prev_hash` の連結検証と `seq` 採番を **live 強制**（`BEFORE INSERT` トリガー）。ハッシュの
  計算（canonical 化 + sha256）は **アプリが所有**する。
  - 代替案 **アプリが全計算・DB は append-only のみ** は却下: 複数ソースが 1 本の global 鎖へ
    並行 append するため、壊れた鎖を live に防げない。
  - 代替案 **DB トリガーが hash まで計算** は却下: canonicalization を PL/pgSQL に抱え、hash
    scheme の進化が重くなる。
- **生バイトは content-addressed な `observation.snapshot`**（`content_hash` を PK にすることで構造的に
  不変）。自前ミラー時は圧縮 `bytes`、非ミラー時は `external_ref`（Internet Archive 等）。event は
  `content_hash` で FK 参照し、**同一トランザクションで挿入**して「event はあるのに snapshot が
  無い」を排除する。これは概念 §14.3 の未決を **「既定はミラー + 例外的に external_ref」**で確定する。
  - 代替案 **外部 object store** は却下: ground truth が 2 システムに跨り、ファイル欠落が連鎖を
    黙って壊す。代替案 **IA 依存のみ** は却下: ground truth の保持が第三者に依存する。
- **不変性は多層防御**: アプリロールから `UPDATE / DELETE / TRUNCATE` を REVOKE し（注: 専用
  app role の配線は infra follow-up。現状はトリガーが実効的防御）、加えて `observation` 各表に
  `BEFORE UPDATE OR DELETE` トリガーを置き無条件 `RAISE`。訂正は常に新イベントで表現する。
- **署名チェックポイント表 `observation.checkpoint` を今定義**（`root_hash` は当面 `chain_head` の
  log_hash、`alg_version` で `linked-v1` → `merkle-v1` を区別、`signature` は NULL 許容）。
  Merkle tree と Ed25519 署名（Signed Tree Head）は schema 変更なしに後付けする。

`interpretation` の二層構造と diff/分類の配置は [[000002]] に分離して記録する。本決定は passive /
read-only（書込は collector の GET 由来のみ）、append-only hash-chain（§4-3）、plane 分離（§4-4）の
設計原則と矛盾しない。

## Consequences

### Pros

- global 単一鎖 + DB による live 強制で、fork / gap / reorder を物理的に排除できる。
- ground truth が単一トランザクショナルストアに一元化され、FK で snapshot 欠落を構造的に防ぐ。
- schema 単位でポリシーが分離する（`observation` は厳格 append-only、`control` は自由可変）。
- `checkpoint` 表により、後から STH / Merkle 検証能力を schema 変更なしに獲得できる。

### Cons / Tradeoffs

- global 鎖は全 insert を単一 writer に直列化する。書込量が穏当なので許容だが、将来の
  ボトルネック候補として残る。
- `chain_head` という可変 1 行を持つ（ただし event 表から再構築可能なカーソルで ground truth では
  ない）。
- REVOKE 層は専用の最小権限 app role がまだ未配線で、owner 接続では無効。実効的防御は当面
  トリガーのみで、role 配線は infra follow-up の負債として残る。
- 概念 §8 の「組み込み・ゼロ依存」と乖離し、Postgres へ依存する。
- 全文の自前ミラーは NDL の DB 著作権（§7）と緊張し得るため、`external_ref` で回避する運用判断が
  個別に要る。
- ハッシュ対象列（canonical 化仕様）をアプリと verifier で version 同期し続ける規律が要る。
