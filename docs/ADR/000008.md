# ADR-000008: M4 を記名投票の選挙区 choropleth として実装し、議員名簿を新Source・地図を現会期スコープに画定する

## Status

Accepted

## Date

2026-06-03

## Affected Services

- api (Go) — `GetVoteEvent` の `Vote` に `house` / `district_code` / `is_pr` / `pr_block` を**読取時 join**（`vote ⨝ legislator_district`）で追加。議案セレクタ用に `ListVoteEvents`（現会期・記名投票のみ）を追加。新 read model `legislator_district` を投影する projector を持つ
- collector (Go) — 新 Source `giin-roster`（両院公式議員名簿）を HTTP GET し `observation` へ append、`legislator_district` を project
- db (Postgres) — `interpretation.legislator_district` 読モデル＋`control` の `giin-roster` source seed
- web (SvelteKit) — MapLibre による選挙区投票地図、国土数値情報 GeoJSON の静的下地、比例パネル、カバレッジ注記、出典明記

## Context

概念 §11 は M4 を「**MapLibre に選挙区/自治体レイヤー**」と定義する。M3 までで横断タイムライン＋議員記名投票記録（[[000006]]）＋Atom フィード（[[000007]]）が実装済みである。

探索で確認した事実 — **現データモデルに地理キーが一切存在しない**:

- `interpretation.person` は `name` / `yomi` / `identity_confidence` のみ。選挙区も都道府県も持たない。
- `interpretation.organization` は会派（`parliamentary_group`）のみ。地理的な党組織は無い。
- M1 国会会議録の `Speech` は `speaker` / `speaker_group`（会派）/ `speaker_position`（役職）だけで、**議員の選挙区は国会会議録 API に無い**。
- M2 e-Gov 法令は**国法令**でサブナショナルな地理スコープが無い。
- リポジトリ全体で `選挙区` / `prefecture` / `electoral` / `geojson` のヒットはゼロ。

つまり地図に乗せる「観測イベント由来の地理」が無い。地図を意味あるものにする唯一の地理束ねは「**選挙区 → その代表議員 → その議員の記名投票**」だが、二つの緊張を解く必要がある: (1) [[000006]] がタイムラインに person 軸を置かないと決め per-person 軸を記名投票記録に限定したこと、(2) **比例選出議員**が地図から構造的に脱落し §5（非党派・選択的監視をしない）と緊張すること。

一方 [[000006]] 自身が「**記名投票の per-person 軸**」をわざわざ許可し、その根拠を「投票者は全員 Diet 議員＝説明責任を負う公職者であり、私人 doxxing リスクが無い」「**自分の代表がどう投票したか**は説明責任の核」と明記している。したがって**公職者の投票の地図化は §4 が禁ずる私人プロファイリングではない**。守るべき線は doxxing ではなく、(a) data 層でスコア化・論評しない（§3 / §5-C）、(b) 脱文脈の差分・スニペットを結論にしない（§7）、(c) 比例を消さずカバレッジを開示する（§5）、に移る。M4 でこの三線を満たす地図の形を確定する必要が生じた。

## Decision

**M4 を「ある1件の記名投票（`VoteEvent`）を、各選挙区の現職議員の賛否という事実カテゴリで塗る選挙区投票地図（district vote map）」として実装する。議員→選挙区の対応は両院公式議員名簿を新 Source として観測面に取り込み、地図は現会期スコープに画定する。**

- **色は事実カテゴリのみ**: 賛成 / 反対 / 棄権 / 欠席 / 記録なし。集計スコア・賛同率・忠誠度は作らない（§3 / §5-C）。区クリックで議案名＋会議録＋既存 `ListLegislatorVotes`（[[000006]]）へ着地する（§7）。議案セレクタで別の記名投票へ切替える。
- **地理は両院**: 衆＝小選挙区（289、議員と 1:1、単色）／参＝選挙区（都道府県単位、1:N、**生カウントの内訳**＝賛成n/反対m。率や「賛成多数」の一色塗りはしない）。合区（鳥取・島根 / 徳島・高知）も扱う。
- **議員→選挙区の源は両院公式議員名簿**: 新 Source `giin-roster` として HTTP GET し観測面 Stream（`ResourceObserved` / `ResourceChanged`）に載せ、`interpretation.legislator_district`（`person_id` → `house` / `district_code` / `is_pr` / `pr_block` / 会派、provenance 付き）へ projector で投影する。制度自身が自らの構成を公示する**公的一次**（§4-2）。**現職のみ**を扱う。
- **時点整合は現会期スコープ**: 地図は現職名簿＋現行区割りで正しく塗れる記名投票（直近総選挙以降）に限定する。過去の投票は**不変ログ／Timeline が保持し続ける**（**地図＝現在のレンズ、ログ＝歴史の記録**）。これにより境界の時点版管理も不要になる。
- **境界は事実でなく地図の下地**: 国土数値情報（国土交通省）の GeoJSON を web の静的アセット（topojson / mapshaper で簡略化）に置き、UI に出典明記する（§9）。事実（person→区・投票）だけが api から provenance 付きで来る（geometry と facts の分離）。
- **§5 開示**: 比例選出議員はその議案で記名投票した分を地図の隣に会派別パネルで**常設併記**し地図から消さない。色凡例に「記録なし区」を含め、「**記名投票議案のみ地図化**」を明示注記する。
- **クエリ表面**: `GetVoteEvent` の `Vote` に `house` / `district_code` / `is_pr` / `pr_block` を読取時 join で追加し、議案セレクタ用に `ListVoteEvents`（現会期・記名投票のみ）を追加する。**地図 read model を実体化せず**、web が GeoJSON join と比例パネルを行う（[[000006]] の薄RPC・読取時合成・リードモデル独立性を継承）。

検討した代替案と却下理由:

- **per-person 選挙区 choropleth を集計スコア（賛同率 / 忠誠度）で塗る**: data 層の評価＝§3 / §5-C 違反。→ 却下。事実カテゴリ単色に限定。
- **時点版管理（会期別名簿＋改定別区割りで歴史も塗る）**: 正確だが、選んだ「現職のみ」源に歴史名簿が無く歴史 GeoJSON も要り、M4 を大きく超え源選択と矛盾。→ 却下。現会期スコープを採用し、歴史は Timeline に委ねる。
- **議員→区の源に Code for Japan / Popolo / Wikidata**: 既に Popolo 形式で interop（§5 / §12）に魅力があるが二次情報で §4-2「公的一次のみ」に抵触。→ 却下。両院公式名簿（一次）を採り、Popolo クロスウォークは将来に留保。
- **境界を watched Source 化**: 完全に on-model だが大きな GeoJSON を RPC で運ぶ不自然さ＋滅多に変わらぬ reference に過剰。→ 却下。静的下地。
- **専用 `GetVoteDistrictMap` RPC / マテリアライズド地図 read model**: 地図ロジックが api・契約に結合、または リードモデル独立性を破り reproject 順序依存を生む（[[000006]] が却下した道）。→ 却下。`GetVoteEvent` 拡張＋読取時 join。
- **比例を出さない（区選出のみ）**: 最小だが §5 と正面違反（比例選出議員と比例依存政党を構造的に脱落）。→ 却下。常設パネル＋注記。

本決定は passive / read-only（名簿も GET のみ）、append-only hash-chain（地図は観測面に書き戻さない解釈面 projection、§4-3）、plane 分離（境界＝下地、事実＝provenance 付き、§4-4）、反 doxxing（公職者の事実記録に限定、私人を扱わない、§4 / DISCIPLINE §4）、コンテキスト保全（色は結論にせず議案＋会議録へ、§7）、非党派（比例を消さずカバレッジを開示、§5）、AI は判定しない（色は機械的な事実カテゴリで評価しない、§4-6）の設計原則と整合する。

## Consequences

### Pros

- §11 M4 を、[[000006]] の説明責任の核（自分の代表がどう投票したか）を地理で実現する形で、規律（§3 / §5 / §5-C / §7）を満たして成立させる。
- 議員→選挙区が新 Source として観測パイプラインに載り provenance を持つ（解釈面の全フィールドが provenance を持つ規律と整合）。
- 現会期スコープにより境界の時点版管理を回避し、地図 read model を実体化せず [[000006]] の薄RPC・読取時合成・リードモデル独立性を保つ。
- 比例パネル＋カバレッジ注記により「地図が何を映さないか」を構造的に開示する。

### Cons / Tradeoffs

- 過去の記名投票は地図に出ない（Timeline／ログが保持）。時点トラベルは後続マイルストーン。
- 衆は起立採決が多く記名投票が疎なため、地図化できる議案が限られる（参の方が記名投票記録が豊富）。「記録なし」が広く出る議案がある。
- 現職のみのため、名簿更新（改選・補選・会派異動）の鮮度を `giin-roster` の再観測で保つ運用規律が要る。
- 新 Source（議員名簿）にも DISCIPLINE §1 のソース別レート制御・robots・識別可能な User-Agent を適用する必要がある。
- 国土数値情報 GeoJSON の出典明記（§9）と、区割り改定時の静的アセット差し替え運用が要る。
- AKN eId と同様、議員名簿側の選挙区コードと国土数値情報側の区コードを**同一に対応づける規律**が要る（drift すると塗れない）。共有契約として明文化・テストで固定する。
