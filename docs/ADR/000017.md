# ADR-000017: 情報密度をデザインの中核から外し画面幅・ポインタ種別に適応する表現と位置づけてモバイル対応する

## Status

Accepted

## Date

2026-06-06

## Affected Services

- web (SvelteKit) — `tokens.css` に `--bp-*` ブレークポイントトークン追加、container クエリ優先のレスポンシブ化（topbar・フィルタ・タイムライン行・投票地図ページ）、`pointer: coarse` でのタップ標的拡大、地図 (`DistrictVoteMap` / `SangiinVoteMap`) の cooperative gestures 化・ポップアップのダークトークン配色・高さ `clamp()` 化
- web (Playwright) — 360px の mobile プロジェクト追加と `responsive.spec.ts`（横スクロール 0・タイムライン可視・フィルタ開閉・地図/nav 到達）
- docs/design/DESIGN_LANGUAGE.md — §8 に標的サイズ項目、§9「レスポンシブ／適応設計」を追加

## Context

ダッシュボードは現在ほぼデスクトップ前提で組まれている。レスポンシブ指定は投票地図詳細の `@media (max-width: 880px)`（2 カラム→1 カラム）1 箇所のみで、他の画面は固定 `max-width` コンテナ＋ `padding: 24px` だけ。360px 幅では次の破綻がある:

- タイムライン行 `ChangeLogItem` は固定 3 カラムグリッド `8.5em 1.5em 1fr`。360px だと時刻列がおよそ 136px を食い、本文（市民の主動線「いつ・何が・どう変わったか」）が約 190px に圧迫される。
- 対話型地図 (`maplibre-gl`) が全幅・固定 `height: 520px`。指を地図に置くとパンがスクロールを奪い、ページを下へ送れない（典型的なスクロール・トラップ）。ポップアップはライト配色（`#15171c` 文字）でダーク中核と不整合。
- ナビ／ページャ／chip リンクが `padding: 4px 10px`・12px で、WCAG 2.5.8（標的サイズ AA, 最小 24px）を下回る。`DESIGN_LANGUAGE.md` §8 の a11y チェックリストにも標的サイズ項目が無い。

ここで根本の緊張は、`DESIGN_LANGUAGE.md` の出自が High-Performance HMI / situation-room、すなわち**大画面・高密度・管制室**の美学である点にある。モバイルは段階的開示と密度低減を要求し、両者は正面衝突する。「デザインの中核を変えずに UI・UX を向上させる」を実装の北極星にするには、**中核とは何か**を先に確定する必要があった。

設計原則との関係: 本決定はプレゼンテーション層のみに閉じ、passive / read-only・append-only hash-chain・plane 分離・「AI は判定しない」には触れない。非党派（CORE_CONCEPT §5）には触れる — 1 カラムに畳む際、比例選出議員（選挙区を持たず地図に乗らない）を画面外に追いやると「地図から消さない」要請に反するため、ここだけは明示的に制約を課す。

## Decision

**情報密度をデザインの中核（意味論）から外し、画面幅・ポインタ種別に応じて適応する「表現」と位置づける。** 全幅で凍結する中核は次に限定する: ダーク既定／色は状態のためだけ（＋グリフ＋ラベルの多重符号）／純黒・純白なし／provenance 常時表示／パネルが情報の最小単位／数値・ID・ハッシュは mono・tabular。`DESIGN_LANGUAGE.md` §1 の「高密度でも疲れにくい」は HPHMI の**デスクトップ表現**であり、狭幅では段階的開示でゆるめてよい。色・テーマ・状態の意味は決して作り替えない。

実装機構:

1. **container クエリ優先・トークン化** — 幅閾値を `--bp-*` として `tokens.css` に集約（下限 360px、360 / 390 で実機検証）。再利用されるパネル／カード／タイムライン行は `@container` で*自身の幅*に応じて折る（「パネルが情報の最小単位」§6 と一致）。ページ骨格（topbar・地図グリッド）だけ最小限の `@media`。
2. **タイムライン行は狭幅で縦積み** — 状態ノード（色＋グリフ＋ラベル）＋時刻を上部メタ行に集約し、タイトル／本文／chip／provenance を全幅で下へ。広幅は現行 3 カラム維持。
3. **フィルタは狭幅で `<details>` 開閉**（既定で閉、`summary` に現在の絞り込みを要約）。主動線を即座に見せる。
4. **ポインタ適応の標的サイズ** — `@media (pointer: coarse)` でタッチ端末のみ操作要素を ≥44×44px（WCAG 2.5.5 相当）。マウス (`pointer: fine`) は現行の高密度を維持。
5. **地図は cooperative gestures** — 1 本指＝ページスクロール／2 本指＝パン・ピンチズーム（ヒントは日本語ロケール）。高さを `clamp()` で狭幅は縮め、ポップアップをダークトークン（`--surface-3` / `--text-1`）で配色。1 カラムに畳む際は比例選出を地図直下に件数アンカー付きで併記し画面外に出さない（CORE_CONCEPT §5）。
6. **検証** — Playwright に 360px の mobile プロジェクトを追加し、`responsive.spec.ts` で横スクロール 0・タイムライン可視・フィルタ開閉・地図/nav 到達をガード（ADR-000016 の再現可能性方針に整合）。

### 検討した代替案と却下理由

- **密度も中核とする**（situation-room の高密度をモバイルでもそのまま維持、許すのはカラム折り畳みとスクロールのみ）: 最も保守的だが 360px で 3 カラム行が確実に窮屈になり、主動線の可読性を落とす。「中核を変えずに UX を向上」の目的を達せない。却下。
- **breakpoint 下で自由に再構成**（ハンバーガー nav・地図詳細を bottom sheet・フィルタ既定非表示の独自モバイル動線）: トークン・色さえ流用すればよいとする案。実装重で、デスクトップの「1 画面統合」思想からの乖離が最大。3 ナビ項目にハンバーガーは過剰。却下。
- **@media 一本**: 既存 880px に揃える単純案。だが同一パネルがメイン／側カラムで異なる文脈に置かれるため、ビューポート幅だけでは「パネル自身の幅」に追従できない。container クエリを主とし @media は骨格に限定する方が §6 と整合。却下。
- **流動・クエリレス優先**（clamp / minmax / flex-wrap で breakpoint 自体を最小化）: 分岐は減るが、高密度グリッド（3 カラム行）が中間幅で半端に崩れやすい。container クエリの離散的な再配置の方が密度の意味を保てる。却下。
- **標的サイズ一律 44px**: 最も快適だがマウス環境のデスクトップ密度を膨らませ HPHMI アイデンティティと衝突。ポインタ適応で「desktop 密・touch 快適」を両立する方が選んだ哲学と一致。却下。
- **モバイル検証は手動のみ**: 安いがレスポンシブ崩れの回帰ガードが無く、ADR-000016 の「`git clone && make test` で全再現」思想とずれる。却下。

## Consequences

### Pros

- 「中核＝意味論／密度＝表現」を明文化したことで、今後のコンポーネントが拠るべき判断基準が固定される。モバイル化が場当たりにならない。
- container クエリ主体なので、パネルは置かれたカラム文脈で自律的に折り、再利用しても破綻しにくい（§6「パネルが情報の最小単位」と素直に噛み合う）。
- ポインタ適応により、デスクトップの高密度（HPHMI）とタッチの快適さ（≥44px）を同一コードベースで両立。WCAG 2.5.5(AAA) 相当をタッチに限って達成。
- cooperative gestures でスマホのスクロール・トラップを解消しつつ地図の対話性を保持。ポップアップ配色のダーク統一は既存バグの解消も兼ねる。
- 360px の Playwright プロジェクトが横スクロール等の回帰を自動検出。`make e2e` の再現可能性に乗る。

### Cons / Tradeoffs

- container クエリは古いブラウザで未対応（baseline 2023〜）。下限ブラウザ方針を別途明示する必要があり、未対応環境では @media フォールバックの検討が残課題。
- タイムライン行が幅で 2 形状（広幅 3 カラム／狭幅縦積み）になり、コンポーネントの CSS 分岐とスナップショット的検証コストが増える。
- cooperative gestures のヒント・2 本指操作はデスクトップのマウス操作と挙動が変わり、`pointer` 種別ごとの体験差を許容することになる（意図的だが、ドキュメントで周知が要る）。
- Playwright に mobile プロジェクトを足すと e2e のラン時間が増える（journey × 2 ブラウザ相当）。
- `DESIGN_LANGUAGE.md` のトークン定義（§2）と新 §9 のブレークポイントトークンが二重管理になり得る。`tokens.css` を単一の真実とし doc は参照に留める運用が必要。
