# ADR-000002: 解釈面を二層化し diff・分類を観測面に焼かず reproject-safe にする

## Status

Accepted

## Date

2026-06-03

## Affected Services

- db (Postgres 18 / Atlas) — `interpretation` を二層（append-only `interpretation.event` + disposable read model）に
- differ (Rust) — diff / 分類 / confidence を `interpretation.change` へ投影
- api (Go) — projector が `observation` + `interpretation` を fold して read model を再生成

## Context

解釈面は「観測面から再計算可能な projection」であり、いつでも DROP → replay できる
disposable な read model にする、というのが原則（CORE_CONCEPT §4-4）。

ところが 2 つの緊張がある:

1. **人手レビュー結果**（administrative / substantive の確定、手動訂正）は観測ログから導出
   できない**新情報**である（§9.3, §11）。これを read model に直接書くと、rebuild で消えて
   reproject-safe が崩壊する。
2. 概念 §9.1 の**イベントスキーマ案**は `diff` / `confidence` を観測イベントの内部フィールドとして
   描いていたが、DISCIPLINE §3「解釈（分類）を観測イベントに書き戻さない」と衝突する。diff は
   2 つの snapshot の純関数（recomputable）であり、不変ログに焼くと差分アルゴリズムの version が
   ground truth に固定され、再投影できなくなる。

§9.1 は表題どおり「案」、§4-4 は「原則」であり、原則を優先して案を締め直す必要がある。

## Decision

**解釈面を二層に分離する。**

- **Tier 1 — `interpretation.event`（append-only）**: 人手・手動の durable な事実（`ReviewVerdict` /
  `Correction` / `Override`）。観測イベントへ `observation_seq` で provenance を持つ。rebuild で
  消してはならない。**独自のハッシュ連鎖**（`interpretation.chain_head`、observation とは別チェーン）を
  かけ、「S4rCiv がレビュー判定を後から書き換えていない」を第三者検証可能にする。
  - 代替案 **append-only + seq のみ（連鎖なし）** は弱い理由で却下: 非党派 watchdog の信頼性に
    連鎖が効き、機構は ADR-000001 から流用できて低コスト。
- **Tier 2 — read model（disposable）**: `interpretation.change` ほか。projector が観測イベント +
  解釈イベントを fold して再生成し、`interpretation.projector_offset` で進捗を追う。mutation-prevention
  はかけず、自由に truncate できる。
- **diff / 分類 / confidence は観測面に焼かず解釈面へ置く**（§9.1 案を §4-4 原則で締め直す）。
  紛らわしい `confidence` を 2 つに割る: **抽出品質**（OCR か否か = 取得時の機械的事実）は
  `observation.snapshot.was_ocr`、**分類 confidence** は `interpretation.change.class_confidence`。
- **read model の再生成は in-place**（truncate → offset reset → replay）。
  - 代替案 **blue-green / versioned projection** は却下: M0 にはオーバー。read model は disposable
    なので、ゼロダウンが要る公開期に schema ロックインなしで後から重ねられる。
  - 代替案 **projection 行の版管理** は却下: disposable 原則と矛盾し、storage / query が膨らむ。
- 標準別 entity（Akoma Ntoso / Popolo / OCDS、vote / contract / funding / timeline）テーブルは
  各 adapter 設計（M1 / M2）に委ね、本 ADR では `change` read model と `projector_offset` のみ確立する。

この決定は plane 分離（§4-4）、AI は判定せず要約のみ（§4-6）、provenance + confidence 必須（§9.2）の
設計原則と矛盾しない。あわせて概念 §9.1 のエンベロープ図を改訂し、`diff` / `confidence` を除いた。

## Consequences

### Pros

- reproject-safe: durable な事実は両プレーンとも append-only event、disposable なのは read model だけ。
- DISCIPLINE §3 を遵守し、観測面／解釈面の分離の純度を保つ。
- 差分アルゴリズム更新が ground truth を汚さず、再投影で反映できる（differ_version を read model に保持）。
- 解釈イベントの連鎖により、人手判定の改竄耐性も担保される。

### Cons / Tradeoffs

- 全テーブルで「append-only event か disposable read model か」を意識し続ける規律コストが生じる
  （immutable-design-guard の監査対象そのもの）。
- projector が 2 種のイベントを正しい順序で fold する複雑さを負う。
- in-place 再生成中は read model が一時的に不整合（`rebuilding` フラグか短時間ダウンで対応）。
  大規模化すると full replay が重くなり、その時点で blue-green への移行が必要になる。
- 概念 §9.1 の図を改訂したため、初稿エンベロープを参照する外部資料とは表記がずれる。
