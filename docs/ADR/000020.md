# ADR-000020: Design Language を記録簿（エディトリアル）方向の v1 に刷新する

## Status

Accepted

## Date

2026-06-06

## Affected Services

- web (SvelteKit) — `tokens.css` の v1 トークン集合化、フォントのセルフホスト配線、コンポーネント restyle、§10 コピー反映
- docs — `DESIGN_LANGUAGE.md` の v0→v1 全面改訂（本 ADR と対）

## Context

`docs/design/S4RCIV`（刷新案プロトタイプ）は、Design Language v0 の**中核を変えずに表現を磨く** v1 案を、ランタイム切替の探索として提示した。中核（無彩色基調・色＝状態のみ＋グリフ＋ラベルの多重符号・ダーク既定・純黒純白なし・provenance 常時・パネルが情報の最小単位・数値は mono tabular・WCAG 2.2 AA）は不変項として凍結されている。

プロトタイプは探索ゆえに複数の可変軸（方向 `記録簿 ledger ⇄ 司令室 room`、トーン `calm ⇄ vivid`、密度）をランタイムトグルで同居させていた。**出荷プロダクトと単一の正典 `DESIGN_LANGUAGE.md` は、これらを 1 つの体系へ確定**しなければならない。確定にあたり、プロダクトの自己記述との緊張が表面化した。

- `CLAUDE.md` / `CORE_CONCEPT` は本プロダクトを「situation-room dashboard（司令室）」と呼ぶ。素直に読めば方向 `room`（運用画面・サンセリフ・丸い角）に寄る。
- しかし**設計原則①は「受動的 sentinel であって actor ではない」**。「situation room」は緊急性・即応・行動を含意し、この受動・非判断の態度と相反する。プロトタイプ自身も方向の既定を `ledger` に置き、表紙コピーを「記録簿の品位を、現代の画面に」とし、エディトリアルな静けさ（公報・新聞・記録簿の品位）を志向していた。

加えて、現状の実装には看過できないギャップがあった。

- **フォントが一切ロードされていない。** `tokens.css` は `IBM Plex Sans JP / Mono` を宣言するが `@font-face` も CDN link も Fontsource パッケージも無く、実際は system フォントで描画されていた。v1 のタイポグラフィ（特に新規の `Zen Old Mincho` 明朝）採用は、フォント供給方式の新規決定を伴う。
- プロトタイプはフォントを **Google Fonts CDN** からロードしていた。これは訪問者の IP を Google に送出し、設計原則①（受動・第三者へ能動的に関わらない）および「単一の自己ホストバイナリ」という目標アーキテクチャと相反する。
- `Zen Old Mincho` / `Plex Sans JP` は CJK フォントで weight あたり数 MB あり、ページ重量への実コストがある。

## Decision

**Design Language を v1 に刷新し、方向 `記録簿 (ledger)` を唯一の正系とする。** 可変軸はランタイムトグルとして出荷せず、単一体系へ焼き込む。具体は以下。

1. **方向 = `記録簿 (ledger)` 正系。`司令室 (room)` は採らない。** 角丸は鋭く（`--r-sm 2 / --r 3 / --r-lg 4`）、構造は影でなくヘアライン罫と余白で作る。`room`（丸い角・サンセリフ見出し・浮くカード）は `DESIGN_LANGUAGE.md` に「採らなかった代替案」として記述のみ残し、出荷 CSS には含めない。**根拠**: 受動・非判断・「淡々と記録する見張り」の態度は、運用画面の即応性より公報/記録簿のエディトリアルな静けさに宿る。プロダクト語の「situation-room」は比喩であり、視覚言語の正系を規定しない。

2. **タイポグラフィ — セリフ（`Zen Old Mincho`）は editorial の一瞬に限定する。** Display／ページ H1／レコードタイトル（法令名・議案名・会議録名）のみセリフ。H2/H3 と画面内見出しは `IBM Plex Sans`、パネル見出しは mono 大文字（§6 既定）、数値・ID・ハッシュ・ラベルは `IBM Plex Mono` tabular。CJK 明朝を密な UI の小サイズに広げると可読性が落ち、フォント重量も増すため、セリフは 600/700 のみ供給する。

3. **状態色トーン = `calm` 単一。** 彩度を抑えた calm を唯一の状態パレットとし（principle #5「警告色は彩度を抑える」と整合）、v0 の彩度（`vivid`）は「採らなかった v0 彩度」として記述のみ残す。`[data-tone]` トグルは出荷しない。

4. **フォントはセルフホストする（weight-controlled）。** 第三者へのリクエストを行わず（IP 送出なし）、`@fontsource`（または静的 `@font-face`）で供給する。`font-display: swap`、weight を絞り（mono 400/500/600、sans JP 400/500/600/700、serif 600/700）、明朝ロード前は system 明朝へフォールバックして初回描画を阻害しない。Google Fonts CDN は採らない。

5. **コピーはレイヤード方針。** 生活者向けの prose／CTA は「確かめる」「改ざん検知用の値」など平易語を用い、技術正規語の `完全性検証` / `署名チェックポイント検証`（`CONTEXT.md` の ubiquitous language）は保持する。**`CONTEXT.md` は変更しない。** マストヘッドの terse ラベル `▸検証` は据え置く。

6. **ランタイム軸は `[data-theme]`（dark/light）のみ残す。** `dir` / `tone` / `density` は焼き込み、出荷では切替不可とする。新トークン一式（`--surface-inset/-raise`、`--hairline-3`、`--accent-2/-line/-on-accent`、`--text-faint`、`--s1..8`、`--fs-*`、`--r-pill`、`--rule`、`--row-pad/--panel-pad`、`--bp-xl`）を採用する。

### 検討した代替案と却下理由

- **方向 `司令室 (room)` を正系にする**: プロダクト語「situation-room dashboard」に文字どおり整合し、新規セリフ依存も不要。→ **却下**。運用画面の即応的トーンは設計原則①（受動 sentinel）と相反する。比喩語に視覚言語を従わせない。
- **ハイブリッド（ledger ＋ 地図のみ room）**: 地図/ダッシュボードだけ運用的質感にする。→ 却下。角丸・見出し体系が 2 系統になり、WCAG 整合の検証面と CSS 表面積が増える割に、地図に独立した運用的アイデンティティを与える必要性が薄い。
- **両方向をランタイムトグルで出荷**: プロトタイプの探索性をそのまま残す。→ 却下。正典が二義的になり、テスト・スクリーンショット・a11y 検証が両系統で必要になる。探索の役目は終わり、確定が要る。
- **Google Fonts CDN**: プロトタイプ通りで最も簡便、意図どおりの明朝が常に出る。→ 却下。訪問者 IP の第三者送出と runtime 依存が原則①・自己ホスト目標に反する。
- **system フォントのみ（system 明朝でセリフ代用）**: 0 バイト・0 リーク。→ 却下。セリフのレンダリングが OS ごとに揺れ（macOS≠Windows≠Linux）、記録簿のアイデンティティが保証されない。weight-controlled なセルフホストで重量を抑えつつ一貫性を取る。
- **状態色 `calm + vivid` トグル**: 彩度をランタイムで選べる。→ 却下。第二の色軸を WCAG 両系で検証する負担に見合わない。calm を単一とする。

本決定は passive / read-only（表示の刷新であり収集・データ層に触れない）、append-only / plane 分離（観測面・解釈面の意味論を変えない）、AI は判定しない（価値含意を持つ色・語を引き続き避ける）と矛盾しない。

## Consequences

### Pros

- 視覚言語が単一の正典に確定し、「なぜ明朝体か／なぜ situation-room と呼ぶのに記録簿の佇まいか」に本 ADR が答える。
- フォントがセルフホストで初めて実ロードされ、宣言と実描画の乖離が解消する。第三者 IP 送出が無く、自己ホストバイナリ目標と整合。
- calm 単一・セリフ限定で、WCAG 検証対象とフォント重量を最小化できる。
- ランタイム軸が `[data-theme]` のみに収斂し、テスト・a11y 検証の組合せ爆発を避ける。

### Cons / Tradeoffs

- CJK フォント（Plex Sans JP / Zen Old Mincho）のセルフホストはページ重量を増す。`font-display: swap`・weight 限定・subset で緩和するが、system フォント時代より重い。
- `司令室` 方向と `vivid` トーンを将来欲した場合、記述からの復活には再びトグル相当の実装と WCAG 再検証が要る（意図的に高くした再導入コスト）。
- プロトタイプ（React/Babel）と出荷（SvelteKit）の二重メンテは行わない。プロトタイプは設計探索の凍結物として `docs/design/` に残す。
