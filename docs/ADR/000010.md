# ADR-000010: kokkai が衆の個人別投票を持たないため選挙区投票地図を参議院記名投票結果に軸足を移す

## Status

Accepted（[[000008]] の「データ源＝kokkai／院＝衆」前提を改める。choropleth の設計規律は継承）

## Date

2026-06-03

## Affected Services

- collector (Go) — 参議院本会議投票結果（touhyoulist）アダプタ＋参議院議員名簿アダプタを新設（予定）
- db (Postgres) — 参の per-member vote 読モデルと参 roster（議員→選挙区）読モデル（予定）
- web (SvelteKit) — 地図を参選挙区（都道府県・1:N の賛否内訳）レイヤーへ（予定）
- api (Go) — 参投票のクエリ表面（予定）

## Context

[[000008]] は M4 の選挙区投票地図を「kokkai の**記名投票者名** → person_id → 選挙区」で塗ると決めた。M4 実装後、実データで検証したところ**この前提が成立しないと判明した**。

**実証（手元データ）**: 第217回 衆議院本会議 2025-03-04（令和7年度予算）の会議録を収集したところ、当該採決は確かに記名投票だったが、会議録には**集計のみ**が記録され、**個人別の賛否氏名は無い**:

> この採決は**記名投票**をもって行います。賛成の諸君は白票、反対の諸君は青票を…
> 〔参事氏名を点呼〕**〔各員投票〕**
> **投票総数 四百五十九／可とする者（白票）二百五十八／否とする者（青票）二百一**

「〔各員投票〕」はト書きで、誰が白票/青票かの氏名リストは会議録（NDL kokkai API）に載らない。他議案は全て「〔賛成者起立〕」＝起立採決で、これも氏名なし。

**Web 調査で裏付け**: 衆議院は起立採決が大半で記名投票は稀（議長判断か出席議員1/5要求時のみ）、かつ個人別結果を公開していない。一方**参議院は押しボタン式の記名投票を常用し、本会議投票結果を議員別にサイト公開**している。M1 の vote parser は合成 fixture（「賛成者 山田太郎…」）で作られ、**実 kokkai データの氏名有無を未検証**だったのが根因。

したがって **kokkai は衆の個人別投票を提供せず、衆 289 小選挙区 choropleth は kokkai データでは原理的に塗れない**。

## Decision

**選挙区投票地図のデータ源を kokkai 会議録から参議院本会議投票結果（touhyoulist）へ移し、地図を参選挙区（都道府県単位・1:N の賛否内訳）レイヤーとして実装する。[[000008]] の choropleth 設計規律（事実カテゴリのみ・集計スコア化しない・比例を消さない・第三者タイル不使用・出典明記・現会期スコープ）はそのまま継承する。**

参議院投票結果の**把握した仕様**（効率的取得の根拠）:

- **索引**: `https://www.sangiin.go.jp/japanese/touhyoulist/{回次}/vote_ind.htm`。回次別（最新 221回＝令和8/2026 現会期）に約50件/会期の記名投票を列挙。
- **個別投票ページ**: `…/{回次}/{回次}-{MMDD}-v{NNN}.htm`（**UTF-8・clean HTML5**、Shift_JIS 不要）。一部 `.pdf`（首相指名等）は対象外。
- **構造**: 会派別（`<dt class="party">`）にまとめ、議員ごとに `<li class="giin"><span class="pros">賛成</span><span class="cons">反対</span><span class="names">議員名</span></li>`。総数は `<h3 class="tohyosousu">`。
- **効率**: **1ページ＝1議案の全議員ロールコール**（実証: 令和8年度予算 245名を1リクエストで取得、賛成126/反対119が `<li class="giin">` 抽出と完全一致）。kokkai の会議単位ポーリングより桁違いに少リクエスト＝レート制御に優しい。
- **選挙区の欠落**: 投票ページは議員名＋賛否＋会派のみで**選挙区を持たない**。都道府県へ写すには**参議院議員名簿**（`/japanese/joho1/kousei/giin/{回次}/giin.htm`、選挙区＝都道府県／比例＝全国）との氏名 join が要る。

検討した代替案と却下理由:

- **衆議院サイトから個人別投票を取る**: 衆議院は個人別記名投票結果を機械可読公開していない（公開の確実な源が無い）。→ 却下。
- **kokkai のまま集計のみ地図化**: 集計（投票総数/可/否）は per-district でないので choropleth を塗れない。→ 不可。
- **衆 choropleth を維持し名前付き投票を待つ**: 衆の名前付き個人別投票は公開データに存在しないため到来しない。→ 却下。

本決定は passive / read-only（touhyoulist も公開ページの GET）、公的一次（参議院自身の公示）、§5（比例＝全国区を別パネルで併記）、§7（色は事実カテゴリ・議案へリンク）、現会期スコープ（[[000008]]）と整合する。

## Consequences

### Pros

- 地図が**実データで塗れる**ようになる（参は個人別記名投票を公開）。
- 取得が効率的（1ページ＝全ロールコール）。レート制御負荷が小さい。
- 参ページが UTF-8 clean HTML5 で、Domino 衆名簿（[[000008]] 実装で foster-parenting に遭った）より堅牢にパースできる。

### Cons / Tradeoffs

- 院が衆→参に変わり、地図ジオメトリが 289 小選挙区（1:1 単色）→ 都道府県選挙区（1:N の賛否内訳）になる。[[000008]] で実装済みの衆 289 choropleth・giin-roster（衆名簿）アダプタは**当面棚上げ**（衆の個人別データが公開されれば復活余地）。
- 参 vote と参 roster の**2アダプタ新設**＋氏名 join（同名異人の扱い）＋参 GeoJSON（都道府県・合区）が要る。M4 を参向けに作り直す規模。
- 議員名のみで選挙区が無いため roster join 必須。氏名表記揺れ（全角空白等）の正規化が要る（既存 PersonIdentity を流用）。
- 合区（鳥取・島根／徳島・高知）と比例（全国区）の描画を別扱いする必要。
