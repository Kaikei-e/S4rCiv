# ADR-000018: ヘッダーを「来歴マストヘッド」として再設計し DESIGN_LANGUAGE を拡張する

## Status

Accepted

## Date

2026-06-06

## Affected Services

- web (SvelteKit) — `+layout.svelte` への共通ヘッダー移設、`+page.svelte` のヘッダー除去、時刻表示の JST 化
- docs/design (`DESIGN_LANGUAGE.md`) — §6 に「マストヘッド」コンポーネントを追加、グリフ語彙の規律を明文化

## Context

現行ヘッダー（`web/src/routes/+page.svelte:51-61`）は以下の点で `DESIGN_LANGUAGE` から逸脱しており、見た目の質が他のコンポーネント（パネル・タイムライン・provenance chip）に追いついていない。

1. **絵文字 `🗺 🗺 📡`** をナビに使用 → §1「無彩色を基調にし、色は状態のためだけに使う」に反する。絵文字はプラットフォーム間で字形・彩度が不一貫で、状態色の語彙とも衝突する。
2. **`◉` をブランド点に流用** → `◉` は §3.3 で「観測（info）」の**状態グリフ**として予約済み。意味を持つグリフを装飾に使うのは符号の過負荷であり、§4「色だけに依存しない＝色＋グリフ＋ラベルの多重符号」の前提を崩す。
3. **H1 への詰め込み** — 「S4rCiv： サーシヴ 公的記録の可視化・追跡」に英名・読み・タグラインを未分化に同居させている。
4. **ヘッダーが各ページに個別実装**され `+layout.svelte` に無い → 全画面での一貫性（運用画面の基本）と再利用が効かない。
5. タグライン「可視化・**追跡**」の「追跡」は能動・監視的な語感で、設計原則①（受動 / sentinel であって actor ではない）および反ドキシングの精神と齟齬する。`CONTEXT.md` は S4rCiv の行為を一貫して「**観測**」と呼ぶ。

刷新方針は外部リサーチ（2026-06-06 実施）で裏付けた。収束した原則は次の3つ。

- **High-Performance HMI / ISA-101**: 最上段は全画面で例外なく見える「ステータス帯」。無彩色基調、色は異常時のみ。ナビ位置は全画面で不変。
- **GOV.UK Design System**: サイト識別（header）とサービス内ナビ（service navigation）の役割分離。ナビはサイトマップでなく上位導線のみ。
- **USWDS banner**: 公式・安全であることを全ページ常設で「装飾でなく機能」として伝える。利用者がそのマーカーを覚えることが目的。

ただし S4rCiv は政府サイトでも産業制御室でもない。原則をそのまま当てると本質を外すため、**オリジナリティの核**として次の翻案を採る。

- **USWDS banner の逆張り**: 政府 banner は「公式だから信じて」と*自称*する。S4rCiv は監視する側であり、信頼を自称せず「**検証可能性**」を提示する（信頼の方向が逆）。
- **HPHMI ステータスの受動的翻案**: 制御室の「機械の状態」に当たるのは、S4rCiv では観測パイプラインの来歴（直近署名チェックポイント・監視カバレッジ）。ただし「collector 稼働中」のような能動・稼働保証の含意は出さない。

## Decision

**ヘッダーを「来歴マストヘッド（provenance masthead）」として `+layout.svelte` に再設計し、`DESIGN_LANGUAGE` §6 にコンポーネントとして追加する。** 視覚はハイパーミニマリズムの mono ワードマーク＋テキストナビとする。

1. **役割** — サイト識別＋来歴の常設帯。GOV.UK 的なサイト識別を担うが、ナビ項目が少数（衆院 / 参院 / 購読）のため**単層**とし、別個の service-navigation 層は設けない。
2. **視覚** — ブランドは mono 大文字ワードマーク「**S4RCIV**」。状態グリフ `◉` の流用を廃止し、ブランドは状態語彙と独立させる。ナビは mono テキストラベル（衆院 / 参院 / 購読）で**絵文字を全廃**する。
3. **第2段（来歴帯）** — スタンス行「観測した事実を記録し、判断はしない」＋ 監視カバレッジ ＋ 直近署名チェックポイント `seq` と `▸検証` 動線。チェックポイントの算出・公開は [[000019]] に従い、データが無い間は当該要素を表示しない（嘘の値を出さない）。
4. **配置** — `+layout.svelte` へ移設し全ページで不変にする。`+page.svelte` 等からヘッダー実装を除去する。
5. **タグライン用語整合** — 「公的記録の可視化・追跡」を「**公的記録の観測ログ**」に改める。`CONTEXT.md` の「観測」に整合させ、能動語「追跡」を避ける。
6. **時刻の JST 化** — 観測時刻・取得時刻を `Asia/Tokyo` で明示フォーマットし「JST」を併記する。`ChangeLogItem.svelte:23` の `(...).replace('T',' ').slice(0,16)` は RFC3339 UTC 文字列を素のまま切り出しており TZ 変換が無い（バグ）。`Intl.DateTimeFormat('ja-JP', { timeZone: 'Asia/Tokyo', ... })` を共通ヘルパに集約し、SSR / ハイドレーション間で安定させる（実行環境 TZ に依存させない）。データジャーナリズムの「時刻とタイムゾーンを明示」原則に従う。

### 検討した代替案と却下理由

- **状況ステータスバー（HPHMI 制御室型をそのまま）**: 「collector 稼働中」「chain ✓ 検証済」を前面に出す案。→ **却下**。「検証済」の自称は [[000014]]（完全性検証は連鎖単位であり S4rCiv 自身が緑の検証済みを名乗らない）に違反し、「稼働中」は設計原則①（受動 / sentinel）と齟齬する。
- **静かな識別帯（ブランド＋ナビのみ）**: 最も安全だが、HPHMI / USWDS / データ報道の「常設プロベナンス」という核を捨て、リサーチ知見をほぼ使わない。→ 却下。
- **記録マーク付きブランド / 二行マストヘッド（枠付き chip ナビ）**: マークの意味付けを誤ると再び符号過負荷になり、chip 枠は現状の汎用 pill に近くダサさが残る。→ 却下し、グリフを一切持たない mono ワードマークを採る。

本決定は passive / read-only（表示は読取専用 projection のみ）、plane 分離（マストヘッドは解釈面の表示で観測面に書かない）、AI は判定しない（スコア・序列を出さない）と矛盾しない。

## Consequences

### Pros

- DESIGN_LANGUAGE §1（無彩色・色＝状態）/ §4（mono ラベル）に整合し、ヘッダーが他コンポーネントと同水準の規律になる。
- 「信頼を自称せず検証可能性を提示」という S4rCiv 固有の姿勢がヘッダーに表現され、模倣でないオリジナリティを担保する。
- `+layout.svelte` 集約で全画面一貫＋再利用が効き、HPHMI の「全画面で不変」を満たす。
- 時刻が JST で正しく読め、タイムゾーン併記で誤読を防ぐ。

### Cons / Tradeoffs

- 来歴帯の「直近署名チェックポイント」は [[000019]] の生成器が出来るまで表示できず、それまで第2段はスタンス行＋監視カバレッジに留まる（段階表示）。
- 監視カバレッジ数・直近チェックポイントの取得には web 向けの小クエリ追加が要る（API に `CountWatches` / `LatestCheckpoint` 相当）。本 ADR は表示要件のみを定め、クエリ実装は実装フェーズに委ねる。
- 単層ナビの判断はナビ項目が増えた場合に再評価が要る（将来 service-navigation 層の分離を検討する余地を残す）。
