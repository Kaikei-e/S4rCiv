# ADR-000027: 日本語 Web フォントを unicode-range スライス化し自ホスト・サブセット配信して LCP を改善する

## Status

Accepted

## Date

2026-06-11

## Affected Services

- web (SvelteKit) — 日本語フォントの供給を `@fontsource` ランタイム配信から、unicode-range スライスの自ホスト配信へ切替。`src/routes/+layout.svelte` の font import、`scripts/subset-fonts.mjs`（再生成）、`src/lib/fonts/`（生成物 + OFL 全文）。

## Context

Cloudflare Web Analytics（24h）で公開 Web の Core Web Vitals が **LCP「悪い」88%**（P50 4,512 / P75 6,468 / P90 8,616 / P99 11,928ms）、CLS は 100% 良好、INP はデータ無し。ほぼ純粋な LCP（最大コンテンツ描画）の劣化だった。

推測を避けて実測で切り分けた結果、配信経路は無罪と確定した:

- **TTFB 96ms**（本番への curl 実測）。HTML は **Cloudflare がエッジで Brotli 圧縮済み**（113KB → 12.5KB, `content-encoding: br`）。よって Cloudflare Tunnel（[[000023]]）・SSR・Connect-RPC は LCP の主因ではない。
- 実機相当トレース（Playwright/Chromium、低速 4G + CPU 4x）で真因を特定: 解析対象のホーム/読書ページが、先頭で **~3.7MB の未スライス CJK woff2** を取得していた。`zen-old-mincho-japanese-700` が単独 ~2MB、低速回線で **~18.5 秒間バンド幅を占有**し、critical path を押し下げる。LCP 要素はレコードタイトル（明朝）だった。
- **woff2 はフォーマット自体が Brotli 圧縮**（W3C WOFF2 仕様）。したがって transport gzip / Nginx リバースプロキシ / API 圧縮では 1 バイトも縮まない。**字数削減（サブセット化）だけが有効なレバー**である。

設計原則との関係: S4RCIV は passive / read-only な sentinel であり、閲覧者 IP を第三者へ漏らさないため**フォントもセルフホスト・第三者リクエスト禁止**（DESIGN_LANGUAGE §11, [[000020]]）。Google Fonts CDN 等の外部 on-demand 配信は採れない。一方、明朝（Zen Old Mincho）は §4.1 で「editorial の一瞬＝Display / ページ H1 / レコードタイトルに限定」と意図的に選定された書体であり、安易な撤去は避けたい。

## Decision

**日本語フォントを unicode-range でスライス化し、自ホストでサブセット配信する**ことを採用する。

- **スライス境界**は Google Noto Sans JP の 124 バケット（頻度グルーピング）を再利用し `scripts/noto-jp-ranges.json` に固定する。各フォントの cmap と交差する符号位置だけを `subset-font`（harfbuzz-wasm、純 WASM・ネイティブ依存なし）でスライス化する（`scripts/subset-fonts.mjs`、`fontkit` で cmap 判定）。ブラウザは `@font-face` の `unicode-range` により、**ページに出現する文字のスライスだけ**を取得する。
- **ウェイトは実使用に削減**する: Zen Old Mincho **700**（レコードタイトル/H1, §4.1）、IBM Plex Sans JP **400 + 600**（本文/画面内見出し）。旧来の全ウェイト import（明朝 600/700、sans-jp 400/500/600/700）を廃止。
- **`font-display: swap` を維持**する。LCP 要素（明朝のレコードタイトル）はフォールバック明朝で即描画され（Lab 実測 700ms）、CLS 100% 良好を保つ。スライスが小さいため遅延スワップも短時間で収束する。
- **生成物（343 woff2 スライス + `jp.css`）はリポジトリにコミット**する。[[000016]] の「`git clone && make test` を Docker だけで再現」を最優先し、install ライフサイクル依存（`prepare` 生成 + gitignore）は CI の `--ignore-scripts` で壊れ得るため採らない。再生成は保守スクリプト `pnpm fonts:build`（フォント/字種を変えたときのみ）。

検討した代替と却下理由:

- **Nginx を Web↔API 間に立てる / API を gzip する**: TTFB は既に 96ms。microcaching は高並列時の負荷対策で、~17 req/日 の本サイトには潰す負荷が無い。woff2 は圧縮済みで gzip 効果ゼロ。要塞化済みの `cloudflared`（[[000023]]）にプロセスを増やす割に LCP 改善ゼロのため却下。
- **単一の Jōyō + Jinmeiyō サブセット（1 ファイル/ウェイト）**: ファイル数は少ないが稀少漢字が脱落する。法令名・議案名・人名は開放語彙であり記録簿に不適。スライスなら全字カバーとページ毎 DL 最小を両立できるため却下。
- **システムフォント化（`@fontsource` 廃止、フォールバック明朝に委ねる）**: LCP は最大改善するが、§4 の書体選定（明朝＝editorial の一瞬）を放棄しデザイン言語の意図に反するため却下。

ライセンス確認: IBM Plex（Sans JP / Sans / Mono）と Zen Old Mincho はいずれも **SIL OFL 1.1**。OFL は GPL/AGPL 等への同梱を許諾し、両者とも **Reserved Font Name 宣言が無い**ためサブセット版に元名を使用できる（openfontlicense.org/ofl-faq で確認）。OFL の最小要件（著作権表示＋ライセンス通知＋全文）を満たすため、OFL 全文を `web/src/lib/fonts/` に同梱し、`/attribution` と README に帰属を明記する。

## Consequences

### Pros

- ホーム相当ページ（`/terms`、同条件: 低速 4G + CPU 4x）でフォント転送 **3,819KB → 614KB（-84%）**、`load` 完了 **19.8s → 4.6s**。現場 LCP 悪化の主因（巨大フォントによる帯域飽和）を除去。
- **全字カバーを維持**（稀少漢字も脱落しない）。記録簿の開放語彙に適合。
- セルフホスト・第三者リクエスト禁止（passive sentinel 原則）を維持。CLS 100% 良好を維持。
- ビルド時の外部ネットワーク依存なし（ソースは `@fontsource` devDep、サブセッタは純 WASM）。[[000016]] の再現性を損なわない。

### Cons / Tradeoffs

- **343 個の woff2 スライス（~5.5MB）をリポジトリにコミット**する。スライス合計はソースより大きい（分割の per-file オーバーヘッド）。フォント更新時は全スライスが churn し、再生成は手動（`pnpm fonts:build`）。
- スライス境界を Google の頻度分割に依存。語彙が偏るページではスライス数が増え得る。
- **現場（RUM）の LCP 改善は Cloudflare Web Analytics で数日かけて確認**する必要がある（Lab は良好を実証済みだが field 反映は未確認）。
- latin サブセット（mono / sans latin）は `@fontsource` import を残置。JP のみ自前スライスで、供給経路が 2 系統に分かれる。
