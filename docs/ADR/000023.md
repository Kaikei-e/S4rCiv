# ADR-000023: Cloudflare Tunnel を http2 固定＋TUNNEL_RETRIES=3 で運用し autoheal は採らない

## Status

Accepted

## Date

2026-06-07

## Affected Services

- `cloudflared`（公開 Web 配信トンネル / `compose.tunnel.yaml` overlay） — `TUNNEL_TRANSPORT_PROTOCOL=http2` 固定（記録の backfill）、`TUNNEL_RETRIES=3` 追加、autoheal は不採用

## Context

S4RCIV の公開 Web は Cloudflare Tunnel（`cloudflared`、タグ固定 `2026.5.2`）経由で配信する。
トンネルの運用設定は gitignore された overlay `compose.tunnel.yaml` に置く（host ローカルで
機微なため）。したがって**この設定の決定は ADR / postmortem にしか durable に残らない**。
実際、過去の http2 固定は overlay のコメントにしか残っておらず、2026-06-07 の障害時に
「同じ問題の再発か別か」の切り分けにログ精読を要した（[postmortem 2026-06-07](../postmortem/2026-06-07-cloudflare-tunnel-1033-edge-timeout.md)）。本 ADR はその経緯を記録する。

2026-06-07 のインシデントでは、ホストから Cloudflare edge レンジ（region1=198.41.192.0/24・
region2=198.41.200.0/24）への外向き TCP/7844 が約12分間全滅し、HA 4 接続すべてが
`i/o timeout` になって Error 1033 が出た。`cloudflared` はプロセス無再起動のまま自己復旧
したが、断絶窓のうち**約4分は回線断ではなく `cloudflared` の再接続 backoff の空回り**だった。

この対応は設計原則①（passive / read-only）と整合する。トンネルは配信経路であり、観測面の
ground truth・収集（collector）とは独立で、いずれもこの決定の影響を受けない。設計原則⑦
（ソース遵守・ハードニング）に関わるのは、可用性のためにコンテナのハードニング姿勢
（`read_only` / `cap_drop: ALL` / `no-new-privileges`）を崩さないか、という点である。

cloudflared の再接続 backoff は公式実装（`retry/backoffhandler.go`）上、base 1s を `1<<retries`
で倍々にし full-jitter で待ち、`RetryForever` ゆえ `retries` は `maxRetries` で凍結して無限に
リトライする。既定 `--retries=5` では待ち時間の天井が実 `1<<5=32s`／ログ表示 `1<<6=64s`
となり、スリープ中はプローブしないため、回線が窓の途中で戻っても次のタイマーまで気づかない。

## Decision

### 決定1（backfill）: `--protocol` を `http2` に固定する

`TUNNEL_TRANSPORT_PROTOCOL=http2` を採用する。理由は、以前この経路で QUIC（UDP/7844）が
"no recent network activity" で断続的に全滅しトンネルがフラッピングしたため。既定の
`--protocol auto` は起動時 precheck でしか http2 に落ちず、稼働後の QUIC 劣化には追従しない。
HTTP 配信オリジンなので QUIC の利点（0-RTT 等）は薄く、ホストの sysctl（UDP buffer）も触らず
に済む。公式上 http2 は列挙値かつ `auto` の fallback 先＝サポートされた安定構成であり、失う
のは PQC 鍵共有（QUIC 専用）程度。従来 overlay コメントのみだった本決定をここに記録する。

### 決定2: `TUNNEL_RETRIES=3` で再接続 backoff の天井を縮める

`TUNNEL_RETRIES=3` を採用する。公式仕様上、設定で backoff の天井を縮められる唯一の手段が
`--retries`（環境変数 `TUNNEL_RETRIES`）で、下げると天井が縮む（`3` → 実 `1<<3=8s`／表示
`1<<4=16s`）。これにより回線復帰後の空回りを既定比で約 1/4 に短縮する。`RetryForever` ゆえ
試行回数は無限のままで、トンネルが諦めることはない。代償は再接続プローブ頻度の微増だが、
相手は Cloudflare 自社の edge であり許容範囲。なお即時リセット（`retries=0` への巻き戻し）は
プロセス再起動でしか起きず、設定では実現できない。

### 決定3（却下・保留）: autoheal による health-driven restart は採らない

`/ready` 監視＋自動再起動は**採用しない**。`/ready` は 0 接続で HTTP 503 を返し、
`cloudflared tunnel ready`（PR #1135、`2026.5.2` に実在）で distroless でも healthcheck 可能。
だが Docker の `restart: unless-stopped` はプロセス exit にしか反応せず unhealthy では再起動
しないため、自動再起動には `willfarrell/autoheal` が要る。autoheal は `docker.sock` マウント＝
ホスト root 相当の権限を持ち、`cloudflared` のハードニング姿勢を実質的に崩す。単発・自己復旧
の事象に対しては過剰と判断し、設定一個で済む決定2 に留める。**再発する場合は、`/ready`
検知と autoheal をまとめて再評価する**（保留）。

### 検討して却下した他の選択肢

- **`--edge-ip-version 6`/`auto` による IPv6 フォールバック**: ホストに生きた IPv6 egress が
  あり、かつ v6 edge 経路もブラックホールでない時しか効かない。今回は IPv4 経路のみで失敗
  しており効果が不確実。予防策であって「窓を狭くする」には直接効かないため今回スコープ外。
- **`--region us` 固定**: 公式に可用性向上の記載はなく、狙う anycast endpoint が変わるだけ。

## Consequences

### Pros

- 回線復帰後の再接続が速くなり、同種の一過性 edge 断での 1033 継続時間が縮む（天井 32s→8s）。
- コンテナのハードニング（`read_only` / `cap_drop: ALL` / `no-new-privileges`）と `docker.sock`
  非マウントを維持でき、設計原則①・⑦と整合する。
- http2 固定と `--retries` 低減の根拠が、gitignore された overlay の外（本 ADR・postmortem）に
  durable に残る。次回の切り分けコストを下げる。

### Cons / Tradeoffs

- 再接続プローブ頻度が微増する（相手が Cloudflare edge なので影響は小さい）。
- トンネル断の検知手段が依然「人間が 1033 を踏む」しかない。`/ready`（503）検知は未導入の
  まま（autoheal の是非と一体で保留）。
- `cloudflared` が「再起動必須で stuck」になる稀なケース（cloudflared #724 / #917）は、決定2
  ではカバーできない。その場合は手動再起動が必要。
- 設定は gitignore された overlay にあるため、適用は host ローカルの手作業に依存する
  （他ホストへ自動展開されない）。
