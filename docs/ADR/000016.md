# ADR-000016: 第三者再現可能なテストスイートを compose.test.yaml の実 Postgres・template-DB-per-test・golden/buf-breaking CDC・Playwright E2E で構成する

## Status

Accepted

## Date

2026-06-06

## Affected Services

- services/api (Go) — DB ドライバの integration テスト（`//go:build integration`、template-DB-per-test）、E2E 決定論シーダ（`//go:build e2eseed`、`internal/e2eseed`）、Dockerfile の `test` ステージ
- services/differ (Rust) — Dockerfile の `test` ステージ（`tests/diff_contract.rs` の in-process diff/eId 契約テストをコンテナで実行）
- web (SvelteKit) — Vitest の 2 プロジェクト（`verification`＝[[000014]] verifier CDC / `component`）、Playwright E2E、pnpm 11 の `allowBuilds`（esbuild）
- infra — `compose.test.yaml`（本番 `compose.yaml` への overlay）、ルート `Makefile`（唯一の実行口）、`scripts/check-proto-drift.sh`、GitHub Actions

## Context

本プロジェクトには E2E・CDC・Unit を貫く、第三者が再現実行できるテストスイートが無かった。Go の domain/usecase/gateway は fake で十分に unit テストされていた一方、**DB に触れるドライバは build-verified のみ**で、観測面の append-only・log chain 連続・reproject 安全・trigger 拒否といった**この企画の中核不変条件が一度も実 Postgres に対して検証されていなかった**。web はテストランナー自体が無く、サービス境界（Go collector→Rust differ、web→Go query api）の契約テストも無かった。

加えてローカルホストは Go 1.23 で go.mod の 1.26 を満たせず、Rust は protoc 不在、atlas も未インストールで、**素のホストではテストが一切走らない**。これは欠点ではなく要件を規定する事実だった——`git clone` した第三者（CORE_CONCEPT §13・§5 が想定する、S4rCiv を信用しない検証者を含む）が同じ壁に当たる以上、再現性は「Docker さえあれば 1 コマンド」で担保するほかない。

設計原則との関係: テストは passive/read-only（live ソースを叩かない）、append-only hash-chain（trigger 不変条件を実 DB で検証）、plane 分離（観測面の整合と解釈面の projection を別レイヤで検証）、source compliance（fixture は記録済み、実名を含めない [[000013]] 系の規律）と整合しなければならない。

## Decision

**テストスイートを 4 層（Unit / CDC / Integration / E2E）で構成し、DB に触れる層はすべて本番 `compose.yaml` を継承した `compose.test.yaml` 上の実 Postgres に対して走らせ、`Makefile` を唯一の実行口とする。** 具体的には:

1. **ハーネス**: explicit な `-f compose.yaml -f compose.test.yaml` overlay（自動マージされる `compose.override.yaml` は使わず、テスト配線が通常 `up` に漏れないようにする）。`services/api`・`web`・`differ` の Dockerfile に `test` ステージを足し、test-runner / seed / web-test / differ-test サービスを overlay 側にのみ定義する。ホスト toolchain には依存しない。

2. **Integration（実 Postgres）**: `//go:build integration` で unit と物理分離。隔離は **template-DB-per-test**（`CREATE DATABASE ... TEMPLATE`）を採用する。append-only ログは trigger がグローバル `seq` を振るため、`seq` 連続性・`log_prev_hash` 連鎖・実 COMMIT 可視性を壊さずに `t.Parallel()` 安全にできるのはこの方式だけである（TRUNCATE は ACCESS EXCLUSIVE で並列不可、tx ロールバック隔離は被テストコードの COMMIT を savepoint に変えてしまう——PostgreSQL 公式 / go-txdb）。マイグレーションは vanilla SQL なので、テンプレート DB には Go ハーネスが `db/migrations/*.sql` をそのまま適用する。

3. **CDC**: 共有 `.proto` を buf が両側に codegen している以上、Pact が埋める「独立モデル間のギャップ」は存在しない。したがって **(a) `buf breaking`（既設定 `FILE`）を deployed ref に対して回す、(b) Go↔differ 境界は golden wire ベクター＋in-process 契約（`tests/diff_contract.rs`）、(c) web↔api は Connect in-memory（Go `httptest` / TS `createRouterTransport`）、(d) 手動同期されている 2 つの `diff.proto` ツリーの byte 一致 CI ガード**で構成する。[[000014]] の verifier↔Go marshal バイト一致 CDC（`web/src/lib/verification/verifier.cdc.test.ts`）はこの CDC 層の一員として既に存在し、`hashable_golden.json` を共有 golden とする。

4. **E2E**: Playwright を**実・決定論シード済みの compose スタック**（browser → SvelteKit SSR → Go api → Postgres）に `baseURL` で向ける。`vite dev` を `webServer` で起動する案は採らない（バックエンドの無い空の front-end になる）。シードは本番と同一のハッシュ経路（`EventFacts.LogHash`）を再利用する Go の決定論シーダ（固定 UUID・固定 UTC 秒・`seq` は trigger 任せ・再シード前に `chain_head` を genesis に戻す）で行い、ダミーデータは実名を含めない（架空のプレースホルダのみ）。

5. **実行口と CI**: `make`（`just` ではなく——追加インストール不要で GH runner と貢献者に同一）を唯一の入口とし、GitHub Actions は service containers ではなく compose-in-CI で全層を回す。

検討した代替案と却下理由:

- **testcontainers-go で ephemeral DB**: hermetic で並列安全だが、本番 compose スタック（同一イメージ・同一マイグレーション・同一配線）から乖離しうる別物を立てる。「テストは実装と乖離してはならない」要件に反するため却下。実 DB は `compose.test.yaml` 経由で立てる。
- **Pact（pact-protobuf-plugin）**: plugin バイナリ＋native FFI＋Pact Broker を Go/TS/Rust に持ち込む運用税が高く、Connect ではなく gRPC 面を検証し、Rust verifier が最も未成熟。共有 `.proto` がスキーマ契約を構築的に保証している本構成では Pact が埋める隙間が無いため却下。
- **状態リセットに TRUNCATE 直列 / go-txdb ロールバック**: 前者は並列不可かつ `chain_head` 手動リセットが要る、後者は append パスの COMMIT 可視性を検証できない。append-only ログには不適として却下し、template-DB-per-test を採る。

本決定は passive/read-only（§4-1）、append-only hash-chain（§4-3）、第三者検証可能（§13・§5）、source compliance（§7/§9）と整合する。

## Consequences

### Pros

- 中核不変条件（trigger による `seq` 付与・broken log chain 拒否・stream/snapshot/event/checkpoint の append-only・`vanished_has_no_content`・content chain 連続・interpretation 面の独立 log chain）が**実 Postgres に対して初めて検証される**。fake では不可能だった保証。
- `git clone && make test`（Docker さえあれば）で第三者が全層を再現実行できる。ホスト toolchain のばらつき（Go バージョン・protoc・atlas）に依存しない。
- テストが本番と同一イメージ・同一マイグレーション・同一配線を使うため、「green locally, red in prod」乖離が構造的に縮む。
- CDC が broker・FFI・plugin ゼロで成立し、2 言語の codegen がずれても byte-drift ガード＋`buf breaking`＋golden で検出できる。
- E2E のシードが本番ハッシュ経路を再利用するため、観測ログと provenance が整合し、結果が byte 再現可能（「このタイムライン項目はこの seq に由来」を厳密に主張できる）。

### Cons / Tradeoffs

- template-DB-per-test は 1 テスト 1 DB 生成のオーバヘッドがある（巨大化したら schema-per-test 等への見直し余地）。テンプレートへの接続が残ると `CREATE DATABASE ... TEMPLATE` が失敗するため、ハーネスは適用後に必ず切断する。
- Go ハーネスがマイグレーション SQL を直接適用するため、本番が atlas で行う**版管理・順序・チェックサム照合は再現しない**（得られるのはスキーマのみ）。マイグレーションが vanilla SQL である前提に乗っており、atlas 固有ディレクティブを導入したら破れる。
- pnpm 11 はビルドスクリプト承認を `pnpm-workspace.yaml` の `allowBuilds`（esbuild）に要求し、これが無いと web イメージのビルド自体が落ちる。新規ネイティブ依存を足すたびに allowlist 更新が要る。
- E2E はブラウザバイナリの導入（`playwright install`）と稼働スタックを要し、最も重く最も壊れやすい層。CI では Playwright の browser-deps 導入が必要。
- reproject 安全性（read model を truncate→replay して一致）の検証はまだ未着手で、本スイートは観測面 trigger 不変条件＋query 経路の検証に留まる（projector を駆動する integration は積み残し）。
