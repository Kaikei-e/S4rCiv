# ADR-000019: 署名チェックポイントに C2SP を採用し linked-v1→merkle-v1 で受動公開 witness に対応する

## Status

Accepted

## Date

2026-06-06

## Affected Services

- collector (Go) — `checkpoint` サブコマンド新設（chain_head を署名して `observation.checkpoint` に append）
- api (Go) — 直近チェックポイントの読取 query、署名チェックポイントの受動公開エンドポイント
- db (Postgres) — `observation.checkpoint` の利用（`signature` / `signer_key_id` 充填）、`alg_version` 名称整合
- web (SvelteKit) — マストヘッドの「直近署名チェックポイント seq ▸検証」表示（[[000018]]）

## Context

`CORE_CONCEPT §5` は「S4rCiv 自身のログ改ざん疑義」への対策として「ハッシュ連鎖＋**定期署名チェックポイント**」を要請する。スキーマ上の器は既にある — `observation.checkpoint` は `through_seq` / `tree_size` / `root_hash` / `alg_version` に加え `signature` / `signer_key_id` カラムを持ち（migration `20260603000001`、コメントに「Starts as the linked-list head; root_hash becomes a Merkle root and signature fill in once signing exists」）、読取側も `Signed` / `SignerKeyId` を公開済み（`handler.go`）。

しかし **production はチェックポイントを 1 件も書いていない**。`observation.checkpoint` への INSERT は `eventlog_integration_test.go` のテストのみで、`verification.go` は「In v0 no signing/anchoring job runs, so any row found has signature NULL and Signed=false」と明記する。欠けているのは器でなく**生成器**である。

S4rCiv は **OSS** であり **Cloudflare Tunnel 経由で Web 公開**される。この2点が設計を規定する。

- OSS ゆえ、ソース・検証手順・公開鍵を公開でき、第三者が自前で検証・witness できる。独自フォーマットを発明するより、既存の透明性ログ・エコシステムに乗るほうが利得が大きい。
- Cloudflare Tunnel ゆえ公開受信ポートを持たず、オリジンは CF 経由で露出する。「公開エンドポイント」「鍵の所在」「TLS 終端」の前提がこの構成に縛られる。

外部リサーチ（2026-06-06、一次情報 C2SP / Sigsum）で核心が確定した。

- **鍵の秘匿だけでは §5 の脅威に効かない。** Sigsum の脅威モデルは「攻撃者がログの署名鍵とインフラを掌握」を明示的に仮定し、防御を鍵秘匿でなく **witness cosignature（独立した第三者が append-only consistency を検証して連署）** に置く。S4rCiv 自身がログを改ざんして再署名する split-view 攻撃を防ぐのは、署名鍵の秘匿ではなく witness 群である。
- **checkpoint は C2SP signed-note**（本文3行: `origin` / `tree_size` 10進 / `root_hash` base64 RFC6962 Merkle root、署名は signed-note 系で **Ed25519 可**）。「clients MUST ignore unknown signatures」により**鍵ローテーションと witness 連署を仕様変更なしに**載せられる。
- **真の witness には Merkle が要る。** 公開 witness 網は O(log n) の consistency proof を要求する。連鎖 head のハッシュ（linked）だけでは compact proof を出せないため、witness 網接続は RFC6962 Merkle tree（`merkle-v1`）で初めて開く。
- 署名は TLS / Cloudflare Tunnel と独立。チェックポイントはオリジンで署名され、公開鍵（OSS リポジトリで配布）で誰でも検証できる。CF が TLS を終端しても CF は署名を偽造できない。

## Decision

**署名チェックポイントのフォーマットに C2SP signed-note / tlog-checkpoint を採用し、`alg_version` で linked-v1 → merkle-v1 の二段進化を採る。鍵は Ed25519（マウント secret）、公開は受動公開（pull させる、push しない）とする。**

1. **フォーマット = C2SP** — 独自フォーマットを発明せず、C2SP signed-note / tlog-checkpoint に準拠する。OSS の透明性ログ・エコシステム（`transparency-dev/witness`、`FiloSottile/litetlog`、Sigsum 公開 witness 網）と無料で相互運用するため。
2. **段階進化** —
   - **v1 = linked-v1**: `chain_head`（singleton: `seq` + `log_hash`）を attest する。`through_seq = tree_size = seq`、`root_hash = chain_head.log_hash`、`alg_version = 'proto-linked-v1'`。Ed25519 で署名し `signature` / `signer_key_id` を充填、受動公開する。linked-v1 は RFC6962 Merkle root を持たないため**準拠 C2SP checkpoint としては公開 witness 網に繋がない**（`origin` に独自 alg を明示し、準拠を誤って主張しない）。
   - **merkle-v1**: `root_hash` を RFC6962 Merkle root に、consistency proof エンドポイント（C2SP tile）を備え、公開 witness 網へ接続する。これが §5 の split-view / 自ログ改ざん防御を実際に成立させる終点。**設計は本 ADR で確定し、実装は段階**とする。
3. **鍵管理** — Ed25519。秘密鍵は**マウント secret 経由**（`CHECKPOINT_SIGNING_KEY_FILE`、env に置かない）で `db.go` の `readSecret()` 規律を踏襲する。公開鍵と `signer_key_id` は OSS リポジトリで配布し、第三者が署名検証に使う。鍵ローテーションは `signer_key_id` で吸収する（過去鍵を保持）。
4. **受動公開** — 署名チェックポイントを公開読取エンドポイント（v1 は Atom / JSON、merkle-v1 は C2SP tile）に晒し、witness・アーカイバ・Internet Archive に **pull** させる。S4rCiv は外部へ **push しない**。
5. **生成 cadence** — `collector checkpoint` サブコマンド（既存の `run` / `poll-once` / `reproject` / `discover` に並ぶ）。各 `run` サイクル末に発行し、加えて周期タイマー（floor 例: 毎時）で発行する。**冪等**とし、`chain_head.seq` が直近チェックポイントの `through_seq` から変化していなければ新規行を作らない。
6. **`alg_version` 名称整合** — v1 を `'proto-linked-v1'` とする（[[000003]] の表記に合わせる）。migration コメントの `'linked-v1'` も同表記に揃える。

### 検討した代替案と却下理由

- **能動アンカリング（Internet Archive / transparency log へ push）**: 最も強い証跡化に見える。→ **却下**。S4rCiv からの能動的な外部書き込みは設計原則①（受動・読取専用・writes 禁止・sentinel であって actor でない）に正面衝突する。`§5` の「Internet Archive 経由で第三者証跡化」も pull（取得経由のアーカイブ）の話であり push ではない。
- **独自署名フォーマット**: 単純だが、OSS の witness 網・既存検証ツールと相互運用できず、第三者検証の裾野を自ら狭める。→ 却下し C2SP を採る。
- **v1 を無署名で出す**: チェックポイント行を今すぐ得られ検証の有界化（[[000014]]）は満たすが、§5 の「署名つき」を満たさず「公開コミットメント」の証拠にならない。→ 却下し v1 から署名する。
- **v1 でいきなり merkle-v1**: 最強だが、Merkle 木構築・proof API・検証の新規一式はスキーマが後送りとした部分の前倒しで、生成器導入のリスクを不必要に上げる。→ linked-v1 を先に出し、merkle-v1 へ `alg_version` bump で進む。
- **鍵を env / KMS に置く**: env は `db.go` が「never placed in an env」と定めた既存規律に反する。KMS は「単一の自己ホストバイナリ」という目標アーキテクチャと自己ホスト civic-tech の精神に反するインフラ依存を持ち込む。→ 却下しマウント secret を採る。

本決定は passive / read-only（生成は観測面の読取と署名のみ、外部 push なし）、append-only hash-chain（チェックポイントは attest であり連鎖を書き換えない）、plane 分離（観測面のみ）と矛盾しない。

## Consequences

### Pros

- §5 の「定期署名チェックポイント」を、OSS の標準（C2SP）に乗って相互運用可能な形で満たす。merkle-v1 で公開 witness 網に繋げば、鍵秘匿に頼らず split-view / 自ログ改ざんを防げる。
- 受動公開ゆえ設計原則①を侵さず、Cloudflare Tunnel 構成とも整合する（witness が CF 経由で pull、署名は CF と独立）。
- スキーマ（`alg_version` / 既存カラム）に相乗りするため、v1 に新規 migration は不要。
- 鍵は既存の secret 規律を踏襲し、env 露出のリスクを増やさない。

### Cons / Tradeoffs

- linked-v1 は公開 witness 網に繋がらないため、v1 単体では「S4rCiv 自身による split-view」を*証明的には*防げない（第三者が公開 head をアーカイブすれば retroactively に露見はする）。完全な防御は merkle-v1 ＋ witness 接続まで未達であることを明示して運用する。
- merkle-v1 は Merkle 木・consistency proof・tile 配信・witness 連携という相応の実装を要する（別途のマイルストン）。
- witness は独立性が要件であり、S4rCiv 自身が witness を兼ねても意味が薄い。第三者 witness または既存網への参加が前提となる。
- Ed25519 は PQC 移行（C2SP の ML-DSA-44 推奨化）に対し将来の `alg_version` bump を要する可能性がある。
