# ADR-000013: 法令ノードに号の細分(Subitem)を加え用語定義(Column)を取り込む eId・文連結契約を確定する

## Status

Accepted

## Date

2026-06-05

## Affected Services

- api (Go) — `internal/domain/legislative` の 法令標準XML パーサと `law_node` 投影
- differ (Rust) — `services/differ/src/xmlmodel.rs` の eId キー node map 導出と `diff.rs` の分類
- db (Postgres) — `interpretation.law_node.node_type` の CHECK ドメイン

## Context

ある政令（425CO0000000280 第14条第2項）で「用語が画面に出ない」不具合を起点に、法令標準XML の取りこぼしを調査した。原因は3つに分かれ、いずれも [[000005]] が定めた **eId キー node map を Go 投影と Rust differ で同一導出する契約**に直接触れる。

1. **用語定義号の Column 取りこぼし**: 定義号は「用語」と「意義」を2つの `<Column>` に分けて持つ（`<ItemSentence><Column><Sentence>…</Sentence></Column><Column>…</Column></ItemSentence>`）。Go パーサは `xml:"Sentence"` で `<ItemSentence>` 直下の `<Sentence>` のみを拾うため、Column 配下を1件も取れず `sentence_text=""` になっていた。一方 Rust differ は sink 方式で Column 配下も連結して拾っていた。**同一 eId に対し Go は空・Rust は連結テキストを返しており、契約が既に破れていた**（differ は M2 構築中で実差分未走行のため顕在化していなかった）。
2. **号の細分(Subitem)の欠落**: イ/ロ/(1)(2) を表す `<Subitem1>`/`<Subitem2>` を Go・Rust 双方がノード化していなかった。号の細分は実質規範テキストを持つため、これは単なる表示欠落ではなく、**細分の改正が差分検知から漏れる「沈黙」**（DISCIPLINE「沈黙は情報」/ §データ整合性に反する）を意味した。
3. **別表(Table/AppdxTable)の欠落**: `<TableStruct>` は本則の外（AppdxTable）にあり、行・列モデルの新設を要する。本 ADR の射程外とする。

`interpretation.change`（差分履歴）はこの時点で空であり、契約を変えても churn が出ない最も安いタイミングだった。

## Decision

**用語定義(Column)を取り込み、号の細分(Subitem)を規範ノードとして materialize する。eId スキームと sentence_text の正規形を Go 投影と Rust differ で byte 単位に一致させる契約として確定する。**

- **Column 降下**: `<ParagraphSentence>`/`<ItemSentence>`/`<Subitem{n}Sentence>` の内容モデルは `(Sentence+ | Column+)` の選択であり、`<Column>` 配下の `<Sentence>` まで降下して owned text を回収する。
- **文連結の正規形**: 1ノードが持つ隣接 `<Sentence>` は **全角スペース U+3000** で連結する（用語／意義、本文／ただし書 など複数文ノード全般に一律適用）。`<Sentence>` 内のインライン要素（Ruby/Sub/Sup…）は文書順に区切り無しで連結する。Go は innerxml を文書順にトークン走査、Rust は sink へのテキスト到達順という別実装だが、出力は一致する。
- **Subitem の eId**: `{parent_eid}__subitem{level}_{Num}`（例 `art_14__para_1__item_2__subitem1_2__subitem2_1`）。深さは要素名 `<Subitem{level}>` から機械的に導く。`Num` 欠落時は親内 1 始まりの序数。
- **node_type**: 細分は深さを eId に持たせ、`node_type` は平坦な **`'subitem'`** 1値とする。`interpretation.law_node` の CHECK ドメインに `'subitem'` を追加するマイグレーションを置く（[[000002]] の Tier-2 disposable read model なので既存行に影響せず、次回 reproject で細分行が現れる）。
- **分類**: 号の細分の sentence_text 変化は **substantive** として扱う（differ の `is_normative` に Subitem を含める）。administrative に飲み込まれてはならない。

検討した代替案と却下理由:

- **Column のみ修正（用語定義だけ直す）**: 報告バグは最小リスクで消えるが、号細分の沈黙（substantive 改正の取りこぼし）を残す。→ 却下。
- **eId を汎用 `__subitem_{Num}`（深さ番号を捨てる）**: 親連鎖で深さは一意だが、XML の明示的な深さタグを捨て忠実性が下がる。→ 却下。深さタグ `subitem{level}_` を採用。
- **node_type を `'subitem1'`/`'subitem2'`… と深さ別にする**: UI インデントを node_type 直で決められるが CHECK 値が増える。深さは eId から読めるため不要。→ 却下。UI は eId の `__subitem` 段数からインデントを算出する。
- **文連結を区切り無し（連結のまま、現 Rust 寄せ）**: Rust 本体を触らず Go を寄せるだけで済むが「用語＋意義」「本文＋ただし書」の境界が潰れ可読性を損なう。→ 却下。全角スペースを正規形に採用し両実装を合わせる。
- **別表(Table)まで取り込む**: 差分精度は最も高いが AppdxTable の行・列モデル新設が要り射程過大。→ 今回は見送り（将来 ADR）。

本決定は [[000005]] の「eId 同一導出」契約を Column/Subitem/連結規則まで拡張するものであり、passive/read-only・append-only hash-chain・plane 分離・AI は判定せず機械的に分類、の設計原則と矛盾しない。

## Consequences

### Pros

- 第14条第2項のような定義条の用語・意義が読取モデルに乗り、画面に表示される。
- 号の細分が独立ノードになり、細分単位の改正が差分（substantive）として検知される。沈黙が解消する。
- Go 投影と Rust differ が同一の node map（実データ 425CO0000000280 で167ノード一致）を返し、[[000005]] 契約の整合が回復・強化される。
- `interpretation.change` が空のうちに正規形を確定したため、既存差分履歴の churn はゼロ。

### Cons / Tradeoffs

- **全角スペース連結と Subitem eId を Go・Rust で同一に保つ規律**が新たに加わる（drift すると全複数文ノードが恒常 MODIFIED 化、または細分が node 行に join できない）。契約テストで固定する。
- `<Sentence>` 内インライン要素やインデックスの少ない別法令で、両実装のテキスト一致が崩れないことの継続検証が要る（Go=innerxml トークン走査 / Rust=ストリーム）。エンティティは数値文字参照・定義済み5種を想定。
- 別表(Table/AppdxTable)の本文・差分は依然未取り込み（将来課題）。
- 号の細分の番号は `Num` 属性値（"1"/"2"）で表示し、イ/ロ と (1)/(2) の字面区別は持たない（号の表示と同方針）。
