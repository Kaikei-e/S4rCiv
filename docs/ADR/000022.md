# ADR-000022: 解釈面 read model の reproject を TRUNCATE 廃止の upsert 上書き replay にして読者原子的にする

## Status

Accepted

## Date

2026-06-07

## Affected Services

- api (Go) — apply が merge-safe upsert / per-stream replace で idempotent な 4 projector（`postgres` driver: 法令 / 議事録 / roster / sangiin-vote）の `BeginRebuild` から `TRUNCATE`（roster は scoped DELETE）を外し、offset を 0 に戻して `rebuilding=true` を立てるだけにする。replay は従来どおり upsert / per-stream replace。`Truncate` 系メソッドは明示的な全消去用に残す
- api (Go) — change projector（`ChangeReadModel`）は `ApplyChange` が bare INSERT で `interpretation.change` に `observation_seq` の UNIQUE が無い（PK は synthetic bigserial）ため、delete 無しの replay は change 行を重複させ timeline の diff を二重計上する。本 ADR では delete-then-replay を維持し、reader-atomic 化（`UNIQUE(observation_seq)` 追加＋upsert）は follow-up とする（その rebuild 窓はタイトル既出の行の diff 件数・分類だけを一時的に空にし、本 ADR が狙うタイトルの生 id 露出には当たらない）
- api (Go) — `timeline.go` の `default:` 分岐（read model 行が無い時）で生 stream id をタイトルに入れるのをやめ、タイトルを空にする
- web (SvelteKit) — `ChangeLogItem.svelte` がタイトル未取得時に生 stream id ではなく種別＋「名称未取得」（例「法令（名称未取得）」）にフォールバックする

## Context

トップの横断タイムラインで、egov-law の行のタイトルが `egov-law:325M50010000064` のような**生の stream id** として露出する事象が観測された。市民から見ると意味不明な内部識別子である。

原因は reproject（read model の再投影）の実装にあった。

- `timeline.go` は観測面 `observation.event` を背骨に、`interpretation.legislative_work` / `interpretation.meeting` を `stream_id` で **LEFT JOIN** して人間可読タイトルを得る。該当行が無いと `default:` 分岐で `v.Title = v.StreamID`（生 id）に落ちる。
- reproject は `BeginRebuild` が **`TRUNCATE` を 1 トランザクションで commit した後**、replay を 1 件ずつ別トランザクションで流す。よって **truncate commit 後〜replay 追いつき前は read model が空**になり、その間 LEFT JOIN が空振りして全 egov-law 行が一斉に生 id になる。

`law.go` の projector は `SnapshotBytes != nil` のイベントしか投影しないため、`ResourceVanished`（スナップショット無し）は read model を更新せず、**消失後も最終既知の法令名が `legislative_work` に残る**（消失行は「法令名 ⊘消失」と名前付きで出る）。したがって `default:`（生 id）に実際に落ちるのは、本質的に **reproject 窓**と、projector が一度もその stream の行を作っていない**コールドスタート/投影遅延**だけだった。

ここで重要なのは、`projection.go` が自己申告するとおり**全投影が merge-safe upsert / per-stream replace で idempotent** であり、観測イベントログは **append-only（観測した stream が後から消える事は原理的に無い）**という点である。すなわち read model を空にする `TRUNCATE` は再投影に必須ではなく、空窓はもっぱらこの `TRUNCATE` に由来していた。これは immutable-design の「reproject-safe projector」「disposable read model」「merge-safe upsert」の論点そのものである。

## Decision

**reproject を「TRUNCATE → replay」から「TRUNCATE 廃止の upsert 上書き replay」に変える。`BeginRebuild` は offset を 0 に戻し `rebuilding=true` を立てるだけにし、replay は既存行の上に merge-safe upsert / per-stream replace で上書きする。読者は再投影中も常に値の入った read model を見る（読者原子的）。併せて UI 側の安全網を一段入れる。**

1. **idempotent な 4 projector（法令 / 議事録 / roster / sangiin-vote）の `BeginRebuild` から `TRUNCATE`／scoped DELETE を外す。** offset=0／`rebuilding=true` のみ。replay は seq 順に各 stream の最新へ収束し、各行は再 fold 時にその場で上書きされる。読者が空の read model を見る窓が消える。change projector だけは bare INSERT＋UNIQUE 無しのため delete-then-replay を維持し、reader-atomic 化を follow-up とする（理由は Affected Services 参照）。
2. **`default:` 分岐で生 stream id をタイトルに使わない。** `timeline.go` はタイトルを空にし、生の内部識別子をクライアントへ渡さない。
3. **UI は種別＋「名称未取得」にフォールバックする。** タイトルが空の行は `ChangeLogItem.svelte` で出典（`egov-law` / `kokkai`）から「法令（名称未取得）」「会議録（名称未取得）」を出す。生 id は出さない。これは reproject 原子化後に残る稀なコールドスタート/投影遅延のための安全網であり、DESIGN_LANGUAGE §10（平易・謙虚）に沿って「何の種類の記録か」だけは誠実に伝える。
4. **`Truncate` メソッドは残す。** projector のロジック変更で出力 stream/行が**減る**場合（新ロジックがある stream を投影しなくなる等）は古い行が orphan として残りうる。M1 ではそのケースは稀で、必要時に明示的な全消去で対処する。常時の reproject 経路からは TRUNCATE を外す。

検討した代替案と却下理由:

- **shadow table に全量構築してから 1 トランザクションでスワップ**: 読者は一切空も中間版も見ない最も強い原子性。ただし別テーブル/スキーマ運用と切替の仕組みが要り M1 にはオーバー。→ 今は不採用。将来 read model が巨大化し、replay 中の per-stream 中間版すら見せたくなくなったら再評価する余地を残す。
- **`BeginRebuild` と全 replay を 1 つの巨大トランザクションにする**: 最小変更で読者は旧→新を原子的に見る。ただし全件 replay 中の長時間トランザクション・ロック保持・メモリで将来破綻する。→ 却下。
- **UI だけ直す（`rebuilding` フラグでバナー＋生 id 抑止）**: データ層を触らず済むが、reproject 中は read model が実際に空のままで、タイトルだけでなく差分件数なども欠ける。空窓そのものは消えない。→ 却下（根本原因が残る）。
- **orphan 掃除（rebuild epoch で stale 行を削除）を常時入れる**: append-only ログでは投影対象 stream が消えないため M1 では空振りに近い追加複雑性。→ 今は不採用。projector ロジックが出力を減らす時だけ明示 `Truncate`。

本決定は append-only hash-chain（観測面は不変・append-only という前提にむしろ依拠する）、plane 分離（disposable な解釈面 read model の再生成方法のみを変え観測面は不変）、passive / read-only（外形挙動は変えない）と整合する。immutable-design-guard の監査対象（reproject-safe projector / merge-safe upsert）。

## Consequences

### Pros

- reproject 中も読者が空の read model を見ない。タイムラインに生 stream id が一斉に露出する窓が消える。
- read model が「いつでも truncate→replay できる disposable な projection」であるという原則を、**truncate を常用しない**形で守る。再投影は idempotent upsert の上書きに収斂し、append-only ログという前提に正しく乗る。
- `default:` で生の内部識別子をクライアントへ渡さなくなり、稀なコールドスタート時も「法令（名称未取得）」と誠実に縮退する。DESIGN_LANGUAGE §10／[[000021]] と同じ「内部識別子を一次ラベルに晒さない」方針で揃う。

### Cons / Tradeoffs

- projector ロジックが**出力を減らす**変更（ある stream/行を投影しなくなる）をした場合、TRUNCATE しないと古い行が orphan として残る。M1 では明示 `Truncate` で対処する運用責務が残る（常時 orphan 掃除は入れない）。
- replay 中、ある stream は古い revision から最新へ順に上書きされるため、その stream の事案ページ（現行全文ツリー）は再投影中に一時的に古い revision を見せうる。空を見せるより良いが、per-stream の中間版が一瞬見える点は shadow swap なら消える積み残し。
- `BeginRebuild` の意味が「truncate＋reset」から「reset のみ」に変わるため、コメント/テストの追従が必要（旧挙動を前提にした truncate 起点の不変条件テストがあれば見直す）。
- change projector は本 ADR の対象外（delete-then-replay 維持）で、reproject 経路の reader-atomic 性に projector 間の非対称が残る。`UNIQUE(observation_seq)` 追加＋`ApplyChange` upsert 化（migration＋Atlas rehash）が follow-up として積み残る。
