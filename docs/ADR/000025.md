# ADR-000025: 完全性検証ビューをオプトイン（ボタン起動）の連結チェーン・スパインに再設計し利用規約に端末内計算を開示する

## Status

Accepted

## Date

2026-06-09

## Affected Services

- `web`（SvelteKit, `src/lib/components/VerificationPanel.svelte`） — マウント時自動再計算を廃止し、ボタン起動のオプトイン walk へ。視覚を設計 `screen-verify.jsx` の連結チェーン・スパインに再設計。法令・会議録ページが共有
- `web`（`src/routes/terms/+page.svelte`） — 「§2 端末内での検証計算（任意・オプトイン）」を新設し以降の条を採番

## Context

完全性検証（[[000014]]）は、`HashableEvent`（[[000003]]）から `log_hash` を**読者自身の端末で再計算**し、記録済みの値と一致するか確かめる仕組みである。受け手は §5「自ログ改ざん疑義」で S4RCIV を信用しない第三者であり、価値は「S4RCIV の緑✓を信じる」ことではなく「数字を読者が再現する」ことにある（glossary：**完全性検証**は連鎖単位の性質、**連結ハッシュの提示と再計算は完全性検証として別扱い**、レコード単位の「検証済み」バッジを避ける）。

しかし従来の `VerificationPanel.svelte` は、

1. ページ表示（マウント）時に WebCrypto で**自動的に**全イベントの `log_hash` を再計算し ✓ を即時表示していた。これは「開いただけで✓が出る」＝**S4RCIV が提示するサーバ側の検証済みバッジ**に見え、glossary が避けるとした提示に接近していた。
2. 視覚は平坦な記録リスト（`ol.records`）で、刷新デザイン案の検証画面が描く「記録の連なり」（縦に連結線で結ばれた一本のチェーン・スパイン）になっていなかった。

加えて、訪問者のブラウザで計算を走らせる行為は、日本では**刑法168条の2・168条の3（不正指令電磁的記録に関する罪）**の論点に触れる。Coinhive 事件最高裁判決（令和4年1月20日・逆転無罪）は、無断・無開示の暗号資産マイニングですら「反意図性は認めるが、PC への影響は広告表示プログラムと**有意差なく社会的に許容される範囲**で**不正性なし**」として無罪とし、判決は「**より適切な利用方法（＝開示・同意）が採り得た**」と指摘した。S4RCIV の再計算は開示・低負荷・ユーザー利益（自己検証）のための処理だが、自動実行は「利用者の明示操作なし」という点で、わざわざ Coinhive が無罪となった条件（社会的許容性）に寄せて運用する理由がない。

なお法的な同意根拠として「サービス利用＝みなし同意」だけに依拠するのは、定型約款（民法548条の2）の「表示」要件（契約締結に至る画面までに認識可能な状態にする等）に照らして弱い。免責は「法律上認められる範囲で N 円まで」式のサルベージ条項が消費者契約法の観点で危険視されており、既存の軽過失限定免責（利用規約 §6＝故意・重過失を除き免責）はこれを避けたクリーンな形である。

## Decision

次の 3 点を一体で採用する。

### 決定1: 完全性検証の再計算を「ボタン起動のオプトイン」にする

`VerificationPanel.svelte` から**マウント時の自動再計算を廃止**し、利用者が「この端末で検証する」ボタンを押したときにのみ `verifyStream` を走らせる。`phase: idle | running | done` の状態機械にし、未操作時は ✓ を一切出さない（連鎖は「（未計算）」表示）。押下後は各ノードの照合結果を 1 件ずつ明らかにする walk（既定 320ms/件、`prefers-reduced-motion` は即時）とし、締めは**連鎖単位**の文言（「この N 件すべてを、記録どおりにこの端末で再現できました」）にして、レコード単位の「検証済み」バッジには見せない。不一致は ✗ と件数で正直に示す（degrade して嘘の✓を作らない）。`HashableEvent` の再正準化・ハッシュ計算には手を入れない（[[000014]] の CDC golden vector を維持）。

### 決定2: 視覚を設計の連結チェーン・スパインに再設計する

平坦リストを、刷新デザイン案の検証画面が示す「記録の連なり（新しい順）」へ置き換える。各ノードは状態の色＋グリフ（◉観測 / Δ変化 / ⊘消失 / ●復活、`ChangeLogItem.svelte` と同一）＋記録 #seq＋観測時刻（JST）＋改ざん検知用の値（`log_hash`）を持ち、縦の連結線で次ノードと結ぶ。チェックポイントと「仕組み」を `<details>` から可視パネルへ出すが、v0 は署名ジョブ未稼働ゆえ **honest degrade**（覆うチェックポイントが無ければ「未生成・連鎖内整合のみ確認」、`signed=false` は「未署名（v0）」）で、署名済みに見せない（glossary：**署名チェックポイント**を「検証済み」バッジ化しない）。`#verify-{seq}` ディープリンクと per-case 記録番号（[[000021]]）は据え置く。`/verify` 専用ルートは作らず、事案ページ内インラインのまま再設計する（per-case 検証〔[[000014]]〕の据え置きを優先）。

### 決定3: 利用規約に「端末内での検証計算」を開示する

`terms/+page.svelte` に「§2 端末内での検証計算（任意・オプトイン）」を新設し、(a) 目的＝記録の整合性を利用者自身で再現・確認できるようにすること、(b) この端末の中だけで完結し計算のため第三者へ送信しないこと、(c) 暗号資産の採掘や行動追跡ではないこと、(d) 実行は**ボタン押下時のみ**でページを開いただけでは走らないこと、(e) 利用をもって本条の仕組み・目的に同意したものとして扱うこと、(f) 結果は参考情報で無保証であること、を明記する。これは「利用＝みなし同意」一本足に依らず、**開示条項＋明示的なオプトイン操作**の二段構えで同意の実体を確保する設計である。免責は既存の軽過失限定免責（§6）を維持し、サルベージ条項は作らない。

### 検討して却下した代替案

- **自動実行のまま視覚だけチェーン化**: 摩擦は最小だが、「開いただけで✓」がサーバ提供の検証済みバッジに見え（glossary と齟齬）、法的にも明示同意の実体を欠く。原則と整合しないため却下。
- **字義どおりの Merkle 分岐ツリー表示**: 現行の `alg_version` は linked-v1。分岐ツリー（RFC 6962 Merkle）は merkle-v1 が前提で未構築であり、提示だけ先行すると実体と乖離する。線形の連結チェーン・スパインを採用する。
- **`/verify` 専用ルートへ分離**（設計原型の独立画面）: per-case 検証（[[000014]]）と `#verify-{seq}` ディープリンク（[[000021]]）の据え置きを崩すため、インライン据え置きでの再設計に留める。

## Consequences

### Pros

- 「読者が自分の端末で再計算する」が明示操作として可視化され、S4RCIV がサーバ側で✓を名乗る外観を避けられる（§5 / glossary 整合）。
- ページを開いただけでは訪問者の CPU を一切使わない。Coinhive で争点となった「明示操作なし」を構造的に解消し、開示＋オプトインで構成要件（反意図性・不正性）に当たらない姿勢を明確化する。
- 同意の根拠が「利用＝みなし同意」一本足でなく、利用規約の開示＋ボタン押下の二段構えになり、定型約款の表示要件の弱さを補える。
- 視覚が刷新デザイン案の検証画面に忠実になり、状態の多重符号（色＋グリフ＋ラベル）と連鎖の連続性が読み取れる。

### Cons / Tradeoffs

- 検証に 1 アクション（ボタン押下）の摩擦が増える。ただし「あなたが再計算する」という主旨上、この摩擦は機能の本質側にある。
- walk は結果を先に一括計算してから 1 件ずつ提示する演出であり、「計算中…」表示は厳密には演出（実計算は押下直後に完了する）。`prefers-reduced-motion` では即時に倒す。
- チェックポイント・署名は v0 で未稼働のため、可視パネルは当面 degrade 表示が常態。署名ジョブ（[[000019]]）の稼働後に表示が充実する。
- 利用規約は「利用＝同意」を引き続き掲げるが、これは表示要件上、拘束力の主たる根拠にはしない前提（本件は計算がオプトインで害も小さいため、みなし合意の弱さは運営者の不利に直結しない）。
