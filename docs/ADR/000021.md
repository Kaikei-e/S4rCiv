# ADR-000021: 記録番号を横断タイムラインの Provenance chip から外し引用ハンドル＋検証 deep-link を事案ページに集約する

## Status

Accepted（[[000014]] を部分 supersede する。[[000014]] の「事案ページ chip は記録 #seq ↗ を deep-link として出す」部分は維持し、「横断タイムラインの各行 chip にも記録 #seq を出す」前提のみを覆す）

## Date

2026-06-07

## Affected Services

- web (SvelteKit) — `ProvenanceChip.svelte` の記録番号を `verifyHref` がある時だけ（＝クリック可能な検証 deep-link の時だけ）描画し、非リンクの平文 `記録 #seq` 分岐を撤去。`ChangeLogItem.svelte`（タイムライン行）は `verifyHref` を渡さないため記録番号が消える。事案ページ（`laws/[lawId]` / `meetings/[issueId]`）の chip は `verifyHref` 付きで `記録 #seq ↗` を従来どおり deep-link として出す
- api (Go) — 変更なし（`Attribution.observation_seq` / `stream_id` は検証 deep-link 用に引き続き提供）

## Context

トップの横断タイムライン各行に出る Provenance chip が、`記録 #2002` のような**生のグローバル `seq` を非リンクの一次ラベルとして全行に露出**していた。市民から見ると「クリックもできず意味も分からない番号」であり、UI/UX 上の異物になっていた。

コードを追うと原因は 2 段だった。

- `ChangeLogItem.svelte`（タイムライン行）は `<ProvenanceChip attr={item.attribution} />` を **`verifyHref` 無し**で呼ぶ。
- `ProvenanceChip.svelte` は `verifyHref` があれば `記録 #seq ↗`（検証パネルへの deep-link）、無ければ**非リンクの平文 `記録 #seq`** を出す二分岐を持つ。タイムラインは常に後者に落ちる。

一方、事案ページ（`laws/[lawId]`）の chip は `verifyHref={#verify-<seq>}` を渡しており、そこでは `記録 #seq ↗` が ADR-[[000014]] の意図どおり**クリック可能な検証 deep-link＋安定した引用ハンドル**として機能していた。つまり chip は事案ページでは正しく働き、**タイムラインだけが死んだ生 seq を出していた**。

これは設計原則・既存決定とも噛み合っていなかった。

- **[[000014]] §3** は「ADR 番号・checkpoint・hash・proto 等の内部用語を一次ラベルとして表に出さない」と明記している。生のグローバル `seq` も同種の内部識別子であり、それを一次ラベルに晒すのは §3 の精神に反する。
- **DESIGN_LANGUAGE §10.3**（声とコピー）は内部語を一次ラベルに出さないことを求める。
- タイムライン行は**タイトル自体が事案ページへのリンク**になっており、同じ行から検証パネルへ飛ぶ deep-link を別途記録番号として出すのは導線の重複でもある。

なお [[000014]] は代替案として「chip から chain 参照を全撤去する」を一度検討し、「per-row の検証導線を失う」として却下していた。本決定はその却下を**横断タイムラインに限って**部分的に再採択する（事案ページの chip では検証 deep-link を維持するので、per-case の検証導線は失われない）。

## Decision

**記録番号（生のグローバル `seq`）を横断タイムラインの Provenance chip から外し、引用ハンドル＋検証パネルへの deep-link は事案ページ（議事録・法令の詳細）の chip にのみ置く。**

実装上は `ProvenanceChip.svelte` の記録番号を **`verifyHref` がある時だけ描画する**よう一本化する。

1. **`verifyHref` 無し（タイムライン行）→ 記録番号を出さない。** chip は `出典 ↗ · 最終取得` のみになる。行のタイトルが事案ページへのリンクを兼ねる。
2. **`verifyHref` 有り（事案ページ）→ `記録 #seq ↗` を検証パネルへの deep-link として出す。** [[000014]] §2 の挙動を維持。記録番号はここで安定した引用ハンドルとしても機能する。
3. **非リンクの平文 `記録 #seq` 分岐を撤去する。** 「クリックできるが意味不明」でも「クリックできず意味不明」でもなく、検証導線として行動可能な文脈にだけ記録番号を出す。

検討した代替案と却下理由:

- **タイムラインでも記録番号をクリック可能化して残す（verifyHref を配線）**: 最小修正だが、`記録 #2002` という生 seq が一次ラベルである点は変わらず「意味不明」は解消しない。タイトルと重複した検証導線も残る。→ 却下。
- **生 seq を可読な引用子（permalink+anchor のコピー導線等）に置換する**: 引用子の再設計を伴い M1 にはオーバー。検証 deep-link は事案ページ chip に既にある。→ 今は不採用。将来 export/引用 UX を作る時に再評価。
- **chip ごと撤去する**: 出典 permalink＋取得時刻は DISCIPLINE §7/§9 が常時併記を要求する core であり撤去できない。→ 却下。

本決定は passive / read-only、plane 分離（記録番号は観測面 `seq` の引用ハンドルで検証は観測面に閉じる）、source compliance（chip の本体価値＝出典 permalink＋取得時刻は維持）と整合する。

## Consequences

### Pros

- タイムライン各行から「クリックできず意味も分からない生 seq」が消え、chip が市民にとって意味のある `出典 ↗ · 最終取得` に絞られる。[[000014]] §3／DESIGN_LANGUAGE §10.3 の「内部語を一次ラベルにしない」と整合する。
- 検証 deep-link＋引用ハンドルは、実際に検証パネルがある事案ページに集約され、行動可能な文脈にだけ残る。per-case の検証導線は失われない。
- `ProvenanceChip.svelte` の二分岐が一本化され、「`verifyHref` がある時だけ記録番号」という単純な不変条件になる。

### Cons / Tradeoffs

- タイムライン行から特定の記録を `seq` で引用する導線が無くなる（引用したい場合は行のタイトルから事案ページへ入り、そこの chip の `記録 #seq ↗` を使う）。一手増えるが、引用は一覧より個別ページで行う方が自然。
- `Attribution.observation_seq` はタイムライン応答に乗り続けるが UI では未使用になる（検証 deep-link 用に事案ページでは使う）。フィールド自体は残す。
- 平文 `記録 #seq` 分岐の撤去で、`verifyHref` を渡さずに記録番号を出したい将来の文脈があれば再追加が要る（現状そのような文脈は無い）。
