# ADR-000005: 観測収集と構造差分をサービス分離し Connect-RPC 契約の stateless differ を採用する

## Status

Accepted

## Date

2026-06-03

## Affected Services

- api (Go) — 法令収集・`LegislativeWork`/AKN ノード投影・差分 usecase（差分結果を `interpretation.change` へ永続化）
- collector (Go) — 法令 snapshot を `observation` へ append、差分 usecase の駆動
- differ (Rust) — 法令標準XML の構造差分 + administrative/substantive 分類のみを担う **stateless** Connect-RPC サーバ
- db (Postgres) — **Go のみが所有**。differ は DB 認証情報を持たず、直接読み書きしない

## Context

[[000001]] / [[000002]] は差分（diff）を解釈面 `interpretation.change` に置き、その算出を **differ (Rust)** に委ねると決めた。両 ADR の Affected Services は「differ が snapshot を取得し interpretation へ投影」と記し、`compose.yaml` の `differ` サービスも Postgres へ直結している。つまり当初設計は **differ が DB を共有して読み書きする**前提だった。

M2（e-Gov 法令アダプタ）は、このプロジェクトで**初めて実際に diff を算出する**マイルストーンである（M1 kokkai は collect + project のみで `interpretation.change` を一度も書いていない。現状 differ は `loop { sleep }` の stub）。したがって M2 で初めて「差分をどのサービスが・どう算出し・どう永続化するか」を確定する必要が生じた。

その際、当初の **共有 DB 前提を見直す**判断に至った。複数サービスが同一 DB スキーマを直接共有するのはサービス境界を曖昧にし、スキーマ変更が暗黙に他サービスを壊す結合（distributed monolith）を生む。観測ログという単一の真実源を、書き手の異なる 2 サービスが直接触る形は、append-only / plane 分離の規律（§4-3, §4-4）を運用面で守りにくくする。

## Decision

**M2 を収集側（Go）と構造差分側（Rust）の 2 サービスに分け、両者を Connect-RPC で接続する。DB は Go のみが所有し、differ は永続化を持たない stateless な計算サービスとする。**

- **責務分割**:
  - **Go (`services/api`)** — 法令現行本文を取得して `observation` へ append（collect）し、`LegislativeWork` メタ + AKN ノード（条/項/号、現行木のみ）を投影（project）する。さらに**差分 usecase** が stream 内の連続 snapshot を読み、differ を呼び、戻り値を `interpretation.change` に書いて専用 `projector_offset` を進める。
  - **Rust (`services/differ`)** — 法令標準XML の**構造差分（AKN eId キーのノード単位 op）と administrative/substantive 分類**だけを所有する。状態も DB も持たない。
- **契約**: `s4rciv.diff.v1.DiffService.ComputeChange(stream_id, prev_snapshot, curr_snapshot, media_type) → { differ_version, classification, class_confidence, node_changes[] }`。Go は**正規化済み（C14N 済み）法令標準XML の生 bytes** を prev/curr で渡し、Rust はノード変化を返す。Go が `interpretation.change` を書く。**共有 DB は採らない**。
- **ノード同一性の共有契約**: AKN eId（`art_{n}` / `art_{n}__para_{m}` / `…__item_{k}`、附則は接頭辞付き）を **Go（read model 行）と Rust（差分ノード参照）で同一導出**する。これにより differ の出力ノードが Go の AKN ノード行に join できる。
- **ツールチェーン**: proto を `buf` で生成し、Go 側は connect-go クライアントを得る。Rust 側は **prost の型生成 + axum/hyper による Connect unary（protobuf over HTTP）の最小実装**とする。gRPC を挟まず、Connect プロトコルを query api と統一する。
- 検討した代替案と却下理由:
  - **共有 DB で differ が直接読み書き**（[[000001]]/[[000002]] の当初前提）: サービス間をスキーマで密結合させる anti-pattern。→ 却下し RPC 契約に置換。
  - **差分を Go（collect/query サービス）内で算出**: 差分の責務を収集/問い合わせサービスに混ぜる。→ 却下。差分は Rust に隔離。
  - **自律 differ（Go が observation 読取/change 書込の 2 RPC を公開し Rust が pull→push）**: offset 状態が結局 Go の DB に残り、RPC 表面と往復が増える。→ 却下。stateless + Go オーケストレーションを採用。
  - **Rust 側 gRPC (tonic)**: 最も枯れているが differ=gRPC / api=Connect の 2 プロトコル混在になる。→ 却下。**Rust 側 Connect 専用 crate (connect-rust)**: プロトコル統一できるが成熟度が不確実。→ 却下。axum 手書きで Connect unary に統一。

本決定は passive / read-only（収集は GET 由来のみ）、append-only hash-chain（差分は解釈面の disposable read model で観測面に焼かない、§4-3 / [[000002]]）、plane 分離（§4-4）、AI は判定せず機械的に分類（§4-6 / §9.3）の設計原則と矛盾しない。あわせて [[000001]] / [[000002]] の Affected Services が記す「differ が DB へ直接投影」を、M2 以降は本 ADR の Connect-RPC 契約に置き換える。

## Consequences

### Pros

- サービス境界が明確になり、Postgres の所有者が Go 単一になる。スキーマ変更が differ を暗黙に壊さない。
- differ が stateless かつ再現可能（`differ_version` を `interpretation.change` に保持し、差分アルゴリズム更新は reproject で反映）。
- 差分・分類の責務が Rust に隔離され、収集/問い合わせサービスが diff ロジックを持たない。
- プロトコルが Connect に統一され（[[000003]] で導入済み）、gRPC 依存を増やさず単一バイナリ志向と整合する。

### Cons / Tradeoffs

- **AKN eId を Go と Rust で同一に導出し続ける規律**が要る（drift すると差分がノード行に join できず navigability が壊れる）。共有契約として明文化・テストで固定する必要がある。
- 大きな法令標準XML（数 MB 級）が prev/curr で RPC を跨ぐ。Connect unary のメッセージ上限調整が要る場面がある。
- Rust 側 Connect unary を手書きするため、フレームワーク任せより HTTP glue の自前コードが増える。
- 1 マイルストーンで 2 サービス（収集 + differ）を配線・運用する複雑さを負う。`compose.yaml` は differ の DB 認証情報を外し、collector/api に `DIFFER_URL` を渡す形へ変える。
- `interpretation.change` の reproject は Go が連続 snapshot で differ を再呼び出しする（差分の replay が differ への RPC 往復を伴う）。
